# Chapter 6.10: Document Chunking Strategies

## Executive Summary

Before you can embed a document, you must decide **how to split it**. This seemingly mundane preprocessing step has an outsized impact on retrieval quality — arguably more impact than the choice of embedding model. Chunk too large, and relevant information is buried in noise. Chunk too small, and you lose the context needed to understand the content. This chapter covers the major chunking strategies (fixed-size, sentence-based, recursive, semantic), analyzes their tradeoffs with concrete examples, and provides production guidelines for choosing chunk sizes and overlap.

---

## Why Chunking Matters

### The Core Problem

Embedding models have **token limits** (typically 256-512 tokens for most models, up to 8192 for newer ones). Even if a model accepts longer inputs, embedding quality degrades with length because the model must compress more information into the same fixed-size vector. A 768-dimensional vector representing a 50-word paragraph retains more of that paragraph's meaning than the same vector representing a 5000-word document.

But the problem goes deeper than token limits. Chunking fundamentally determines **what unit of information can be retrieved**. If your chunks are full documents, a query about one specific fact in a 10-page report will retrieve the entire report — and the user (or LLM) must find the relevant sentence somewhere in those 10 pages. If your chunks are individual sentences, you might retrieve the exact fact but lose the surrounding context needed to understand it.

### Impact on Retrieval Quality

Consider a product manual with this passage:

```
"The X200 processor supports ECC memory when used with B650 motherboards.
However, when paired with older B550 boards, ECC support is limited to
non-registered DIMMs only. Maximum supported memory speed is DDR5-5200
with the latest BIOS update (version 2.4.1 or later required)."
```

**If chunked as one unit**: A query "does X200 support ECC?" retrieves the full context, including the important B650 vs B550 distinction and the DDR5-5200 speed detail. The answer is complete and nuanced.

**If each sentence is its own chunk**: "The X200 processor supports ECC memory when used with B650 motherboards" is retrieved. The B550 caveat and BIOS requirement are in different chunks and may not be retrieved. The answer is technically correct but dangerously incomplete.

**If chunked in the middle of a thought**: "...with older B550 boards, ECC support is limited to non-registered DIMMs only. Maximum supported memory speed is DDR5-5200" — this chunk starts mid-context (what is B550 referring to? what product?) and ends mid-topic. The embedding will be confused about what this chunk is about.

## Strategy 1: Fixed-Size Chunking

### How It Works

Split text every N tokens (or characters), regardless of content boundaries. This is the simplest approach and the most commonly used baseline.

```python
def fixed_size_chunk(text, chunk_size=512, overlap=50):
    """
    Split text into fixed-size chunks with overlap.
    
    chunk_size: number of tokens per chunk
    overlap: tokens shared between consecutive chunks
    
    The overlap ensures that sentences split at chunk boundaries
    are present in at least one chunk in their entirety.
    """
    tokens = tokenize(text)
    chunks = []
    
    start = 0
    while start < len(tokens):
        end = start + chunk_size
        chunk = tokens[start:end]
        chunks.append(detokenize(chunk))
        start = end - overlap  # Slide window back for overlap
    
    return chunks

# Example: 2000-token document, chunk_size=500, overlap=50
# Chunks: [0:500], [450:950], [900:1400], [1350:1850], [1800:2000]
# → 5 chunks with 50-token overlap between each pair
```

### Advantages

- **Simple and predictable**: Every chunk has the same size, making memory planning straightforward. You know exactly how many chunks you'll get from a document.
- **Fast**: No need for any NLP processing — just count tokens and split.
- **Works with any content**: Doesn't require sentence detection, paragraph markers, or structural analysis.
- **Uniform embedding quality**: All chunks have the same token count, so the embedding model processes consistent input lengths.

### Disadvantages

The fundamental problem with fixed-size chunking is that text boundaries rarely align with meaning boundaries:

- **Splits mid-sentence**: "The server crashed because of a memory leak in the connection pool—" / "—which was caused by unclosed database connections." These two chunks represent one causal explanation, but neither chunk captures the full cause-and-effect relationship.
- **Splits mid-paragraph**: A paragraph explaining a three-step process might be split after step 2, leaving step 3 orphaned in the next chunk without context.
- **Splits mid-table/code**: A table or code block that spans the chunk boundary becomes meaningless fragments in both chunks.

### When to Use

Fixed-size chunking is a good starting point and is often sufficient for:
- Large-scale indexing where processing speed matters more than per-chunk quality
- Homogeneous text (news articles, blog posts) without complex structure
- Quick prototypes before investing in more sophisticated chunking

## Strategy 2: Sentence-Based Chunking

### How It Works

Split text at sentence boundaries, then group sentences until reaching a target chunk size. This ensures no sentence is ever split across chunks.

```python
import nltk

def sentence_based_chunk(text, max_chunk_tokens=512, overlap_sentences=2):
    """
    Split at sentence boundaries, group into chunks.
    
    Never splits a sentence across chunks.
    Overlap is measured in sentences, not tokens,
    ensuring complete thoughts carry over between chunks.
    """
    sentences = nltk.sent_tokenize(text)
    chunks = []
    current_chunk = []
    current_tokens = 0
    
    for sentence in sentences:
        sent_tokens = count_tokens(sentence)
        
        if current_tokens + sent_tokens > max_chunk_tokens and current_chunk:
            # Save current chunk
            chunks.append(' '.join(current_chunk))
            
            # Carry over last N sentences as overlap
            current_chunk = current_chunk[-overlap_sentences:]
            current_tokens = sum(count_tokens(s) for s in current_chunk)
        
        current_chunk.append(sentence)
        current_tokens += sent_tokens
    
    # Don't forget the last chunk
    if current_chunk:
        chunks.append(' '.join(current_chunk))
    
    return chunks
```

### Advantages

- **Preserves sentence integrity**: Every sentence appears whole in at least one chunk. No mid-sentence splits.
- **Sentence-level overlap** is more meaningful than token-level overlap — you're overlapping complete thoughts, not arbitrary token spans.
- **Better embeddings**: Complete sentences produce higher-quality embeddings than sentence fragments.

### Disadvantages

- **Variable chunk sizes**: Some chunks may be much shorter than others (e.g., a section with short sentences followed by a section with long sentences). This can impact retrieval fairness — shorter chunks may have higher cosine similarity to queries simply because they're more focused.
- **Sentence detection isn't perfect**: Abbreviations ("Dr.", "U.S.A."), decimal numbers ("3.14"), and URLs can confuse sentence boundary detectors.
- **Ignores document structure**: A topic shift in the middle of a paragraph creates a chunk that covers two unrelated topics.

## Strategy 3: Recursive Character Splitting (LangChain Default)

### How It Works

This is the most popular chunking strategy in practice, used as LangChain's default. It tries a hierarchy of separators, falling back to smaller separators only when needed:

```python
def recursive_split(text, chunk_size=512, separators=None):
    """
    Try to split at natural boundaries, falling back to smaller ones.
    
    Default separator hierarchy:
    1. "\n\n" (paragraph boundaries) — best semantic separation
    2. "\n"   (line breaks) — good for structured text
    3. ". "   (sentences) — preserve complete thoughts
    4. " "    (words) — last resort before character split
    5. ""     (characters) — absolute fallback
    """
    if separators is None:
        separators = ["\n\n", "\n", ". ", " ", ""]
    
    for separator in separators:
        splits = text.split(separator)
        
        # If splits are small enough, use this separator
        if all(count_tokens(s) <= chunk_size for s in splits):
            return merge_splits_to_target_size(splits, chunk_size, separator)
        
        # If some splits are still too large, try next separator on those
        result = []
        for split in splits:
            if count_tokens(split) <= chunk_size:
                result.append(split)
            else:
                # Recursively split with finer separator
                result.extend(recursive_split(split, chunk_size, separators[1:]))
        
        return merge_splits_to_target_size(result, chunk_size, separator)
```

### Why It's Popular

The recursive approach is pragmatic: it produces the best results "out of the box" for most document types. Paragraphs are usually the ideal chunk boundary (they represent complete thoughts), so trying paragraph splits first captures most of the benefit. When paragraphs are too long, falling back to sentences is a reasonable second choice.

## Strategy 4: Semantic Chunking

### How It Works

Instead of using syntactic boundaries (sentences, paragraphs), semantic chunking uses **embedding similarity** to find natural topic boundaries. Adjacent sentences with similar embeddings stay together; a sharp drop in similarity indicates a topic change and triggers a chunk boundary.

```python
def semantic_chunk(text, model, similarity_threshold=0.75):
    """
    Split text based on embedding similarity between adjacent sentences.
    
    When the similarity between sentence N and sentence N+1 drops
    below the threshold, create a chunk boundary. This captures
    natural topic shifts that syntactic methods miss.
    """
    sentences = nltk.sent_tokenize(text)
    embeddings = [model.encode(s) for s in sentences]
    
    chunks = []
    current_chunk = [sentences[0]]
    
    for i in range(1, len(sentences)):
        # Compare current sentence embedding to previous
        similarity = cosine_similarity(embeddings[i-1], embeddings[i])
        
        if similarity < similarity_threshold:
            # Topic shift detected — start new chunk
            chunks.append(' '.join(current_chunk))
            current_chunk = [sentences[i]]
        else:
            # Same topic — continue current chunk
            current_chunk.append(sentences[i])
    
    if current_chunk:
        chunks.append(' '.join(current_chunk))
    
    return chunks
```

### Advantages

- **Topic-aware boundaries**: Chunks naturally align with topic changes, producing coherent units of meaning
- **Adaptive sizing**: Short, focused topics produce small chunks; extended discussions produce larger ones
- **Best embedding quality**: Each chunk is about one coherent topic, producing focused embeddings

### Disadvantages

- **Computationally expensive**: Requires embedding every sentence individually (O(N) model forward passes just for chunking)
- **Threshold sensitivity**: The similarity threshold requires tuning per domain. Too high → too many tiny chunks. Too low → too few large chunks.
- **Model dependency**: Chunking quality depends on the embedding model, creating a circular dependency (you need embeddings to chunk, but chunking quality affects embedding quality)
- **Not deterministic across models**: Changing your embedding model changes your chunk boundaries, requiring complete re-indexing

## Strategy 5: Structure-Aware Chunking

### How It Works

Leverage document structure (headers, sections, lists, tables, code blocks) to create chunks that respect the document's own organization:

```python
def structure_aware_chunk(document, max_chunk_tokens=512):
    """
    Use document structure (Markdown headers, HTML tags, etc.)
    to create chunks that respect the author's content organization.
    """
    sections = parse_sections(document)  # Split on ## headers, <h2> tags, etc.
    
    chunks = []
    for section in sections:
        header = section.header  # e.g., "## Installation Guide"
        content = section.content
        
        if count_tokens(content) <= max_chunk_tokens:
            # Section fits in one chunk — prepend header for context
            chunks.append(f"{header}\n{content}")
        else:
            # Section too large — sub-chunk by paragraphs within section
            sub_chunks = recursive_split(content, max_chunk_tokens)
            for i, sub in enumerate(sub_chunks):
                # Prepend header to every sub-chunk for context
                chunks.append(f"{header} (part {i+1})\n{sub}")
    
    return chunks
```

### Key Insight: Prepend Context

The most important technique in structure-aware chunking is **prepending the section header** (and potentially the document title) to every chunk. This gives the embedding model critical context:

```
Without context:
  "To install, run pip install faiss-cpu. For GPU support, use pip install faiss-gpu."
  → Embedding: something about pip installation (vague)

With context:
  "FAISS Library > Installation Guide: To install, run pip install faiss-cpu. 
   For GPU support, use pip install faiss-gpu."
  → Embedding: installing the FAISS library (specific and accurate)
```

This context-prepending technique is applicable to ALL chunking strategies, not just structure-aware chunking, and consistently improves retrieval quality by 5-15%.

## Chunk Size Guidelines

### The Goldilocks Zone

Extensive experimentation across multiple domains and retrieval systems has converged on a **sweet spot of 256-512 tokens** for most applications:

| Chunk Size | Precision | Context | Best For |
|-----------|-----------|---------|----------|
| 50-100 tokens | Very high | Very low | FAQs, definitions, single facts |
| 128-256 tokens | High | Moderate | Q&A, customer support, factoid retrieval |
| 256-512 tokens | Balanced | Good | **General purpose (recommended default)** |
| 512-1024 tokens | Moderate | High | Technical docs, research papers, narratives |
| 1024+ tokens | Low | Very high | Long-context models, document summarization |

### Why 256-512 Tokens Works Best

The sweet spot exists because of two competing forces:

1. **Smaller chunks improve precision**: The query "What temperature should I bake pizza at?" needs one specific fact. A 100-token chunk containing just that fact produces a very focused embedding that closely matches the query embedding. A 1000-token chunk containing the entire recipe produces a diluted embedding where the temperature detail is averaged with 50 other facts.

2. **Larger chunks preserve context**: If the chunk says "Bake at 475°F" but the surrounding context says "for thin-crust Neapolitan style only; for deep dish, use 400°F" — you need enough chunk size to capture the conditional. Too small, and you serve incorrect advice.

At 256-512 tokens, you typically capture 1-2 complete paragraphs — enough for a coherent explanation with context, but focused enough for a meaningful embedding.

### Overlap Guidelines

```
Recommended overlap: 10-20% of chunk size

chunk_size=256  → overlap = 25-50 tokens
chunk_size=512  → overlap = 50-100 tokens
chunk_size=1024 → overlap = 100-200 tokens

Why overlap?
- Ensures sentences at chunk boundaries appear in full in at least one chunk
- Prevents information loss at the seams between chunks
- Adds ~15-20% more chunks to index (storage/memory cost)

Too much overlap (>30%):
- Wastes storage (many near-duplicate chunks)
- Can confuse retrieval (multiple chunks return near-identical content)
- Fills the LLM context window with redundant passages
```

## Production Recommendations

### Decision Matrix

| Factor | Fixed-Size | Sentence | Recursive | Semantic | Structure |
|--------|-----------|----------|-----------|----------|-----------|
| Implementation complexity | Low | Low | Medium | High | Medium |
| Processing speed | Fastest | Fast | Fast | Slow | Medium |
| Retrieval quality | Acceptable | Good | Good | Best | Very Good |
| Handles code/tables | Poor | Poor | Medium | Medium | Best |
| Content-type agnostic | Yes | Yes | Yes | Yes | No |
| Recommended first try? | ✅ | — | ✅ | — | — |

### A Practical Starting Configuration

```python
# Start here for most applications:
config = {
    "strategy": "recursive",          # LangChain default — works well OOB
    "chunk_size": 512,                # Sweet spot for most use cases
    "overlap": 50,                    # ~10% overlap
    "separators": ["\n\n", "\n", ". ", " "],
    "prepend_title": True,            # Always prepend doc/section title
    "min_chunk_size": 100,            # Drop chunks smaller than this (noise)
}

# Then iterate:
# 1. Evaluate retrieval quality with this config
# 2. If precision is low → decrease chunk_size to 256
# 3. If context is missing → increase chunk_size to 768
# 4. If structured docs → switch to structure-aware
# 5. If quality-critical → try semantic chunking
```

## Key Takeaways

1. **Chunking has more impact on retrieval quality than model choice** — a mediocre model with good chunks outperforms a great model with bad chunks
2. **256-512 tokens** is the sweet spot for most applications, balancing precision (find the right fact) with context (understand the fact in context)
3. **Recursive splitting** is the best default strategy: tries paragraph boundaries first, falls back gracefully to sentences, then words
4. **Always prepend context** (document title, section header) to every chunk — this single technique improves retrieval quality by 5-15%
5. **Overlap of 10-20%** prevents information loss at chunk boundaries at a modest storage cost
6. **Semantic chunking** produces the highest quality chunks but is computationally expensive and creates a circular dependency with the embedding model
7. **Structure-aware chunking** is ideal for well-structured documents (Markdown, HTML, technical docs) where headers and sections provide natural boundaries
