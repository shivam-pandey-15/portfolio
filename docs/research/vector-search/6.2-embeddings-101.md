# Chapter 6.2: Embeddings 101

## Executive Summary

Embeddings are the bridge between human-readable text and machine-processable numbers. They transform words, sentences, or entire documents into fixed-length vectors of floating-point numbers, where geometric proximity encodes semantic similarity. This chapter traces the evolution from sparse bag-of-words representations to modern dense embeddings, explaining the key architectures, training objectives, loss functions, and practical considerations for choosing and deploying embedding models.

---

## From Sparse to Dense: The Evolution

### Bag-of-Words / TF-IDF (Sparse)

Traditional representations are **sparse vectors** with dimensions equal to the vocabulary size:
- Vocabulary of 100,000 words → 100,000-dimensional vector
- Each dimension corresponds to a word
- Most values are zero (sparse)

```python
# TF-IDF: "the cat sat on the mat"
# Vocab: [cat, dog, mat, on, sat, the, ...]
vector = [1.2, 0, 0.8, 0.3, 0.9, 0.1, ..., 0, 0, 0]  # 100K dims, mostly zeros

# Memory: 100K × 4 bytes = 400 KB per document
# For 1M docs: 400 GB just for vectors (mostly zeros → use sparse format)
```

**Problem**: No semantic awareness. "cat" and "kitten" are as different as "cat" and "airplane." Every word is orthogonal to every other word.

### Word2Vec (2013) — Static Word Embeddings

Google's Word2Vec introduced **dense, low-dimensional** word representations (typically 100-300 dimensions).

**Core Principle**: The **distributional hypothesis** — words appearing in similar contexts tend to have similar meanings. "coffee" and "tea" appear near "drink," "cup," "hot," so they get similar vectors.

**Training Objective**: Words appearing in similar contexts should have similar vectors.

#### Two Architectures

```python
# 1. CBOW (Continuous Bag of Words): Predict center word from context
#    Context: ["the", "cat", "on", "the"]  →  Predict: "sat"
#    Faster training, better for frequent words
#    Window: typically 5-10 words

# 2. Skip-gram: Predict context words from center word
#    Center: "sat"  →  Predict: ["the", "cat", "on", "the"]
#    Slower but better for rare words
#    Creates more training examples per word
```

#### Training Optimization: Negative Sampling

The full softmax over 100K+ vocabulary is prohibitively expensive:

```python
# Full softmax: O(V) per training step — too slow for V=100,000+
# P(word|context) = exp(score(word)) / sum(exp(score(w)) for ALL w in vocab)

# Negative Sampling: binary classification instead
# For each positive (word, context) pair:
#   - Train: this pair IS related (label=1)
#   - Sample 5-20 random "negative" words
#   - Train: these pairs are NOT related (label=0)
# Reduces O(V) to O(k) where k=5-20

def negative_sampling_loss(center, context_word, negative_words, k=5):
    """
    Instead of computing softmax over entire vocabulary,
    treat it as binary classification of positive vs negative pairs.
    """
    # Positive pair: center word + true context word
    pos_loss = -log(sigmoid(dot(center, context_word)))
    
    # Negative pairs: center word + random words (not in context)
    neg_loss = sum(-log(sigmoid(-dot(center, neg))) for neg in negative_words)
    
    return pos_loss + neg_loss
```

**Alternative**: Hierarchical Softmax — organizes vocabulary in a binary Huffman tree, reducing O(V) to O(log V). Better for rare words but more complex.

#### Famous Result: Vector Arithmetic

```python
# Word2Vec learns: king - man + woman ≈ queen
vec("king") - vec("man") + vec("woman") ≈ vec("queen")

# More examples:
vec("Paris") - vec("France") + vec("Germany") ≈ vec("Berlin")
vec("walking") - vec("walk") + vec("swim") ≈ vec("swimming")

# Dimensions: 300 (not 100,000)
# Dense: most values are non-zero
vec("cat") = [0.21, -0.04, 0.67, 0.15, ...]  # 300 dimensions
```

#### Dimension Choice

| Dimensions | Use Case | Quality | Memory per 1M words |
|-----------|----------|---------|---------------------|
| 50-100 | Small datasets (<100K sentences) | Basic semantic capture | 200-400 MB |
| 200-300 | General NLP (Google's default) | Sweet spot for most tasks | 800 MB - 1.2 GB |
| 768+ | Only with massive data | Diminishing returns for W2V | 3+ GB |

**Rule of thumb**: For Word2Vec, 300 dimensions hits the sweet spot. Beyond that, overfitting risk increases without proportional gains.

**Critical Limitation**: One vector per word, regardless of context. "bank" (river) = "bank" (finance). This is called the **polysemy problem**.

### GloVe (2014) — Global Vectors

Stanford's GloVe combined Word2Vec's local context with **global co-occurrence statistics**:
- Builds a word-word co-occurrence matrix from entire corpus
- Factorizes it to produce embeddings
- Often slightly better than Word2Vec for analogies

```python
# GloVe's insight: the RATIO of co-occurrence probabilities encodes meaning
# P(solid | ice) / P(solid | steam) = large  → "solid" relates more to "ice"
# P(water | ice) / P(water | steam) ≈ 1     → "water" relates equally

# Loss function: make dot product of embeddings ≈ log of co-occurrence count
# J = Σ f(X_ij) × (w_i · w_j + b_i + b_j - log(X_ij))²
```

### FastText (2016) — Subword Embeddings

Facebook's FastText extended Word2Vec with **character n-grams**:
- "running" → [<ru, run, unn, nni, nin, ing, ng>]  (character 3-grams)
- Word vector = sum of character n-gram vectors
- Can generate vectors for **out-of-vocabulary** words (e.g., "covidvaccine")
- Better for morphologically rich languages (Finnish, Turkish, Arabic)

```python
# FastText: word "where" (with n=3)
# Subwords: <wh, whe, her, ere, re>
# vec("where") = vec("<wh") + vec("whe") + vec("her") + vec("ere") + vec("re>")

# This means FastText can handle misspellings:
# "whre" shares subwords with "where" → similar vector!
```

## The Transformer Revolution

### BERT (2018) — Contextual Embeddings

BERT (Bidirectional Encoder Representations from Transformers) changed everything:

1. **Contextual**: The same word gets different vectors depending on context
   - "I sat by the river **bank**" → bank_vector₁ (nature-related)
   - "I went to the **bank** to deposit money" → bank_vector₂ (finance-related)

2. **Pre-trained**: Trained on massive corpora (Wikipedia + BookCorpus, 3.3B words)
3. **Fine-tunable**: Adapt to specific downstream tasks
4. **Bidirectional**: Reads text in both directions simultaneously (unlike GPT which is left-to-right)

**Architecture Details**: 

| Variant | Layers | Hidden Size | Params | Training Time |
|---------|--------|-------------|--------|---------------|
| BERT-base | 12 | 768 | 110M | 4 days on 16 TPUs |
| BERT-large | 24 | 1024 | 340M | 4 days on 64 TPUs |

**Training Objectives**:
- **Masked Language Modeling (MLM)**: Randomly mask 15% of tokens, predict them from context
- **Next Sentence Prediction (NSP)**: Given two sentences, predict if second follows first

**Problem for Search**: BERT produces per-token embeddings (one vector per word). To compare two sentences, you'd need to feed them jointly through BERT — this is O(N²) for N documents.

### Sentence Transformers (2019) — Sentence-Level Embeddings

BERT alone produces **token-level** embeddings. For search, we need **sentence-level** vectors.

**Sentence-BERT (SBERT)** by Reimers & Gurevych solved this:
- Uses Siamese/Triplet network training
- Produces fixed-size vectors for entire sentences
- Trained so semantically similar sentences are close in vector space
- **10,000x speedup** over pure BERT for finding similar pairs in 10K sentences (65 hours → 5 seconds)

#### How SBERT Produces Sentence Embeddings

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('all-MiniLM-L6-v2')

# Internally:
# 1. Tokenize input sentence
# 2. Pass through transformer layers → get per-token embeddings
# 3. Apply POOLING to get single vector:
#    - Mean pooling (average all token vectors) ← default, usually best
#    - CLS pooling (use [CLS] token vector)
#    - Max pooling (element-wise max across tokens)

query_vec = model.encode("How to fix a flat tire")      # [384]
doc_vec   = model.encode("Steps for puncture repair")   # [384]

# Cosine similarity: ~0.82 (high match despite zero word overlap!)
similarity = cosine_similarity(query_vec, doc_vec)
```

#### Training Loss Functions

The choice of loss function critically affects embedding quality:

```python
# 1. MultipleNegativesRankingLoss (MNR / InfoNCE)
#    THE dominant loss for search embeddings
#    Input: (anchor, positive) pairs only — negatives come from other pairs in batch
#    All other positives in the batch act as negatives
#    Larger batch size = more negatives = better training signal

def mnr_loss(anchors, positives, temperature=0.05):
    """
    In a batch of N (anchor, positive) pairs:
    - anchor_i should be close to positive_i
    - anchor_i should be far from positive_j (j ≠ i)
    """
    scores = cosine_similarity_matrix(anchors, positives)  # NxN
    labels = range(N)  # diagonal should be highest
    return cross_entropy(scores / temperature, labels)

# 2. TripletLoss
#    Input: (anchor, positive, negative) triplets
#    Requires explicit negative mining
#    margin controls minimum separation

def triplet_loss(anchor, positive, negative, margin=0.5):
    """
    d(anchor, positive) + margin < d(anchor, negative)
    """
    pos_dist = distance(anchor, positive)
    neg_dist = distance(anchor, negative)
    return max(0, pos_dist - neg_dist + margin)

# 3. CosineSimilarityLoss
#    Input: (sentence_A, sentence_B, similarity_score) triples
#    Good for STS (Semantic Textual Similarity) tasks
```

## Training Objectives for Search Embeddings

### Contrastive Learning (Deep Dive)

The dominant training paradigm for search embeddings:

```python
def contrastive_loss(query, positive, negatives, temperature=0.05):
    """
    Push query close to positive (relevant doc),
    Push query away from negatives (irrelevant docs).
    
    Temperature controls how "peaked" the distribution is:
    - Low temp (0.01): very peaked, focuses on hardest negatives
    - High temp (0.1): smoother, more uniform gradient
    """
    pos_score = cosine_similarity(query, positive)
    neg_scores = [cosine_similarity(query, neg) for neg in negatives]
    
    # InfoNCE loss: maximize positive, minimize negatives
    loss = -log(exp(pos_score / temperature) / 
                sum(exp(s / temperature) for s in [pos_score] + neg_scores))
    return loss
```

**Key insight**: Batch size matters enormously. With batch_size=32, each query gets 31 in-batch negatives. With batch_size=1024, it gets 1023. Larger batches → better negative diversity → better embeddings.

### Hard Negative Mining

Simple random negatives are too easy → model doesn't learn fine distinctions.

**Hard negatives**: Documents that are close to the query but NOT relevant:
- "How to train a puppy" (query)
- "How to train a machine learning model" (hard negative — similar words, different meaning)

Mining strategies:
1. **BM25 negatives**: Top BM25 results that are not marked relevant — lexically similar but semantically different
2. **In-batch negatives**: Other queries' positives within same training batch — free, but may be too easy
3. **Cross-encoder negatives**: Use a cross-encoder teacher to score all pairs, pick highest-scoring non-relevant as negatives — most effective but expensive
4. **Iterative mining**: Train model round 1, use its own retrieval to find hard negatives, retrain round 2

```python
# Example: Hard negative mining pipeline
def mine_hard_negatives(query, relevant_docs, index, model, top_k=30):
    """
    Use the model's own retrieval to find tricky negatives.
    """
    query_vec = model.encode(query)
    candidates = index.search(query_vec, k=top_k)
    
    # Keep candidates that are NOT in the relevant set
    hard_negatives = [c for c in candidates if c.doc_id not in relevant_docs]
    
    # These are documents the model thinks are relevant but aren't
    # → Training on these forces the model to learn finer distinctions
    return hard_negatives[:5]  # Top-5 hardest
```

### Fine-Tuning Domain-Specific Models

General models may not understand domain jargon. Fine-tuning fixes this:

```python
from sentence_transformers import SentenceTransformer, InputExample, losses
from torch.utils.data import DataLoader

# 1. Start with pre-trained model
model = SentenceTransformer('all-MiniLM-L6-v2')

# 2. Prepare domain-specific training pairs
train_examples = [
    InputExample(texts=["myocardial infarction symptoms", 
                        "chest pain and shortness of breath during heart attack"]),
    InputExample(texts=["ACL reconstruction recovery", 
                        "rehabilitation timeline after anterior cruciate ligament surgery"]),
    # ... hundreds to thousands of domain pairs
]

# 3. Choose loss function
train_dataloader = DataLoader(train_examples, batch_size=32, shuffle=True)
train_loss = losses.MultipleNegativesRankingLoss(model)

# 4. Fine-tune (typically 1-3 epochs, ~10 minutes on GPU)
model.fit(
    train_objectives=[(train_dataloader, train_loss)],
    epochs=3,
    warmup_steps=100,
    show_progress_bar=True,
)
```

**When to fine-tune**:
- Domain has specialized vocabulary not in general web text
- Retrieval recall on your data is below target
- You have at least 500-1000 training pairs
- General models confuse your domain terms (e.g., legal "consideration" ≠ "thoughtfulness")

## Embedding Properties for Search

### Distance Metrics (Deep Dive)

| Metric | Formula | Range | When to Use |
|--------|---------|-------|------------|
| Cosine Similarity | dot(A,B) / (‖A‖·‖B‖) | [-1, 1] | Default for text. Ignores magnitude, measures angle only |
| Dot Product | dot(A,B) | (-∞, +∞) | If vectors are L2-normalized (equivalent to cosine) |
| Euclidean (L2) | ‖A - B‖₂ | [0, +∞) | Sometimes for images. Sensitive to magnitude |
| Manhattan (L1) | Σ|A_i - B_i| | [0, +∞) | Sparse vectors, binary features |

**Important**: If vectors are L2-normalized (‖v‖ = 1), cosine similarity = dot product. Most embedding models output normalized vectors, so cosine and dot product give identical rankings.

```python
# Normalizing vectors: makes cosine = dot product
import numpy as np

def normalize(v):
    return v / np.linalg.norm(v)

v1_norm = normalize(v1)
v2_norm = normalize(v2)

# These are now equivalent:
cosine_sim = np.dot(v1, v2) / (np.linalg.norm(v1) * np.linalg.norm(v2))
dot_product = np.dot(v1_norm, v2_norm)
# cosine_sim ≈ dot_product
```

### Dimension Reduction Techniques

Higher dimensions capture more nuance but cost more memory and compute:
- 768-dim × 4 bytes × 1M docs = **3 GB** just for vectors
- 768-dim × 4 bytes × 100M docs = **300 GB** → needs multiple machines
- 768-dim × 4 bytes × 1B docs = **3 TB** → requires quantization or reduction

| Technique | How It Works | Quality Loss | Use Case |
|-----------|-------------|-------------|----------|
| PCA | Linear projection | 5-10% at 256-dim | Post-hoc reduction |
| Matryoshka | Train model for multi-dim | 2-5% at 256-dim | State-of-the-art |
| Random projection | Preserve distances via JL lemma | 5-15% | Fast, no training |
| Quantization (SQ) | float32 → int8 | 2-5% | Reduce bytes, not dims |

**Matryoshka Embeddings** (2022) — the modern approach:

```python
# Matryoshka training: model is useful at ANY prefix length
# Full vector: [0.21, -0.04, 0.67, ..., 0.15]  # 768 dims (3072 bytes)
# Truncated:   [0.21, -0.04, 0.67, ..., 0.33]  # 256 dims (1024 bytes)
# Further:     [0.21, -0.04, 0.67, ..., 0.88]  # 64 dims (256 bytes)

# Example: nomic-embed-text-v1.5 with Matryoshka
model = SentenceTransformer('nomic-ai/nomic-embed-text-v1.5')

# Use full dimensions for high accuracy
full_vec = model.encode("query", output_value='sentence_embedding')  # 768-dim

# Truncate for faster search (3x less memory, ~2% quality loss)
truncated_vec = full_vec[:256]  # Just take first 256 dimensions!
```

## Key Takeaways

1. **Sparse → Dense**: Evolved from enormous sparse vectors (TF-IDF, 100K dims) to compact dense vectors (300-1536 dims) that encode meaning
2. **Context matters**: Word2Vec gives one vector per word (static); BERT/Sentence Transformers give contextual vectors that change based on surrounding text
3. **Training**: Modern search embeddings use contrastive learning (InfoNCE/MNR loss) with hard negative mining. Batch size and negative quality are critical
4. **Negative sampling**: Word2Vec uses negative sampling (O(k) vs O(V)); SBERT uses in-batch negatives and hard negative mining for fine-grained distinctions
5. **Dimensions = cost tradeoff**: Higher dims → better quality but more memory/compute. 384-768 is the sweet spot. Matryoshka embeddings allow flexible dimension choice
6. **Fine-tuning**: Domain-specific fine-tuning is essential when general models mishandle specialized vocabulary
7. **MTEB benchmark**: Use MTEB leaderboard to compare models, but benchmark on YOUR domain data for production decisions
