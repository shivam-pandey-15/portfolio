# Chapter 6.4: Vector Indexing Basics

## Executive Summary

Once we have embeddings, we need to actually **find** the most similar vectors quickly. Brute-force comparison is O(N×d) per query—impossibly slow for millions of vectors. This chapter covers the fundamental vector indexing strategies: flat indexes, IVF (Inverted File), PQ (Product Quantization), and their combinations—including advanced topics like Voronoi cell boundary effects, asymmetric distance computation (ADC), Optimized PQ (OPQ), and practical FAISS configuration for billion-scale systems.

---

## The Brute-Force Baseline

### Exact Nearest Neighbor (Flat Index)

```python
def brute_force_search(query_vec, all_vectors, k=10):
    """
    Compare query against EVERY vector in the database.
    Exact results, but O(N × d) per query.
    """
    distances = []
    for i, vec in enumerate(all_vectors):
        dist = cosine_distance(query_vec, vec)
        distances.append((dist, i))
    
    distances.sort()
    return distances[:k]
```

### Why Brute Force Fails

| N (docs) | d (dims) | Distance ops | Time at 1B ops/sec | Verdict |
|----------|----------|-------------|---------------------|---------|
| 10K | 768 | 7.68M | ~8ms | ✅ Fine |
| 100K | 768 | 76.8M | ~77ms | ⚠️ Borderline |
| 1M | 768 | 768M | ~768ms | ❌ Too slow |
| 100M | 768 | 76.8B | ~77 seconds | ❌ Unusable |
| 1B | 768 | 768B | ~768 seconds | ❌ Absurd |

At scale, we need **approximate** methods that sacrifice some accuracy for massive speed gains. The key tradeoffs:
1. **Partition the space** → only search relevant regions (IVF)
2. **Compress the vectors** → smaller, faster distance computation (PQ)
3. **Build a graph** → navigate to neighbors efficiently (HNSW, Chapter 6.5)

## IVF (Inverted File Index)

### Core Idea

Partition the vector space into **clusters** (Voronoi cells) using k-means. At query time, only search the closest clusters.

### Voronoi Cells

Each cluster defines a **Voronoi cell**: the region of space closer to this centroid than any other. Every vector belongs to exactly one cell.

```
┌─────────────────────────────────────────┐
│           ·  ·                          │
│     ·   ·  C1  ·    ·                   │
│   ·   · · ·  ·  ───────  ·  C2  ·  ·   │
│     ·   ·    /    ·  ·     ·   ·        │
│            /    ·               ·       │
│  ─────── /  ─────────────────           │
│         /         ·                     │
│   C3  ·/    ·   ·     C4   ·  ·        │
│    · · ·  ·       ·  ·  ·    ·          │
│     ·   ·           ·    ·              │
└─────────────────────────────────────────┘

C1, C2, C3, C4 = cluster centroids
Lines = Voronoi cell boundaries
· = data vectors assigned to nearest centroid
```

### Construction

```python
def build_ivf_index(vectors, n_clusters=1024):
    """
    Step 1: Cluster all vectors using k-means
    Step 2: Assign each vector to its nearest centroid
    
    Training data requirement: at least 30 × n_clusters vectors
    (e.g., 30K vectors for 1024 clusters)
    """
    # k-means clustering (iterative: assign → recompute centroids → repeat)
    centroids = kmeans(vectors, k=n_clusters, n_iter=20)
    
    # Build inverted file: cluster_id → list of (doc_id, vector) pairs
    inverted_lists = defaultdict(list)
    for i, vec in enumerate(vectors):
        cluster_id = nearest_centroid(vec, centroids)
        inverted_lists[cluster_id].append((i, vec))
    
    return centroids, inverted_lists
```

### Search

```python
def ivf_search(query_vec, centroids, inverted_lists, n_probe=10, k=10):
    """
    Step 1: Find closest centroids to query (n_probe of them)
    Step 2: Search ONLY within those clusters
    
    Speedup ≈ n_clusters / n_probe (e.g., 1024/10 = 100x)
    """
    # Find nearest centroids (fast: compare query to n_clusters centroids)
    nearest = find_k_nearest(query_vec, centroids, k=n_probe)
    
    # Search only in selected clusters (skip 99% of data)
    candidates = []
    for cluster_id in nearest:
        for doc_id, vec in inverted_lists[cluster_id]:
            dist = cosine_distance(query_vec, vec)
            candidates.append((dist, doc_id))
    
    candidates.sort()
    return candidates[:k]
```

### The Boundary Problem

The fundamental weakness of IVF: a query near a cluster **boundary** may have true nearest neighbors in an adjacent cell that isn't probed.

```
        Voronoi boundary
              │
    Cell A    │    Cell B
              │
         q ← │ → nn    ← query's true nearest neighbor is in Cell B!
              │
    nprobe=1: search only Cell A → MISS nn
    nprobe=2: search both A and B → FIND nn ✅
```

**Mitigation strategies**:
1. **Higher nprobe**: Search more cells (diminishing returns above ~5% of clusters)
2. **Multi-probe hashing**: Perturb hash to include boundary neighbors 
3. **Over-clustering**: Use more clusters than needed → smaller cells, fewer boundary issues
4. **Residual refinement**: Compute residual vector (vec - centroid), search residuals for finer precision

### nlist (Number of Clusters) Sizing

| Dataset Size | Recommended nlist | Rationale |
|-------------|------------------|-----------|
| < 1M | 4 × √N (≈ 4K) | Rule of thumb |
| 1M - 10M | 16 × √N (≈ 16K-50K) | Finer partitioning |
| 10M - 1B | 65,536 - 262,144 | Typical production values |
| > 1B | 1M+ | Very fine partitioning |

**nprobe tuning**: Start at nprobe = nlist × 0.01 (1% of clusters), measure recall, increase until target recall is met.

### Centroid Quality and Retraining

k-means centroids must be representative of the data distribution:
- **Training data**: Use at least 30 × nlist vectors (representative sample)
- **Data drift**: If data distribution changes significantly, centroids become stale → recall drops
- **Retraining schedule**: Rebuild index weekly/monthly depending on data velocity
- **OPQ rotation** (see below): Can improve centroid quality by rotating the space first

## PQ (Product Quantization)

### Core Idea

Compress vectors by splitting them into sub-vectors and quantizing each independently. This converts a 3KB float vector into a ~96-byte code while enabling fast approximate distance computation.

### How It Works — Step by Step

```
Original vector (768 dims, 3072 bytes):
[0.21, -0.04, 0.67, ..., 0.15, 0.88, -0.32]
 |─── sub-vec 1 ───|  ...  |── sub-vec M ──|

Step 1: Split into M sub-vectors (e.g., M=96, each has 8 dims)
  sub_1 = [0.21, -0.04, 0.67, 0.11, 0.85, -0.02, 0.44, 0.33]
  sub_2 = [0.15, 0.72, -0.31, 0.99, 0.05, -0.67, 0.28, 0.41]
  ...
  sub_96 = [0.15, 0.88, -0.32, 0.04, 0.67, -0.11, 0.55, 0.22]

Step 2: For each sub-space, learn 256 centroids via k-means
  codebook_1: 256 centroids in 8-dim space
  codebook_2: 256 centroids in 8-dim space
  ...
  codebook_96: 256 centroids in 8-dim space

Step 3: Replace each sub-vector with its nearest centroid ID (1 byte = 0-255)
  PQ code: [42, 187, 5, 220, ..., 73, 18, 199]  ← 96 bytes total
```

### Compression Ratios

| Original | M (sub-vectors) | bits | Compressed | Ratio |
|----------|-----------------|------|-----------|-------|
| 768-dim float32 = 3072 B | 96 | 8 | 96 B | **32x** |
| 768-dim float32 = 3072 B | 48 | 8 | 48 B | **64x** |
| 768-dim float32 = 3072 B | 192 | 8 | 192 B | **16x** |
| 128-dim float32 = 512 B | 8 | 8 | 8 B | **64x** |

Scaling impact:

| Scale | float32 | PQ (M=96) | Savings |
|-------|---------|-----------|---------|
| 1M vectors | 3 GB | 96 MB | 31x |
| 100M vectors | 300 GB | 9.6 GB | 31x |
| 1B vectors | **3 TB** | **96 GB** | 31x |

### Distance Computation: ADC vs SDC

PQ enables two modes of distance computation:

#### Asymmetric Distance Computation (ADC) — Preferred

The query vector stays **uncompressed**. Distances are computed between the full query and the compressed database vectors:

```python
def adc_distance(query_vec, pq_code, codebooks, M, d_sub):
    """
    ADC: query is NOT quantized, database vector IS quantized.
    More accurate because query has full precision.
    
    Pre-computation: O(M × K × d_sub) — once per query
    Per-vector distance: O(M) lookups — very fast
    """
    # Pre-compute: distance from each query sub-vector to all centroids
    # This creates M lookup tables, each with 256 entries
    dist_tables = []
    for m in range(M):
        query_sub = query_vec[m*d_sub : (m+1)*d_sub]
        table = [l2_distance(query_sub, centroid) 
                 for centroid in codebooks[m]]  # 256 distances
        dist_tables.append(table)
    
    # Compute distance: just M table lookups (NOT M×d_sub multiplications)
    total_dist = sum(dist_tables[m][pq_code[m]] for m in range(M))
    return total_dist

# Speed: 96 table lookups instead of 768 floating-point multiplications
# → ~8x faster per distance computation
```

#### Symmetric Distance Computation (SDC) — Faster but Less Accurate

Both query AND database vectors are quantized. Distances computed from pre-calculated centroid-to-centroid tables:

```python
def sdc_distance(query_pq_code, db_pq_code, centroid_distance_tables, M):
    """
    SDC: both query and database vector are quantized.
    Faster (no per-query table computation) but less accurate.
    
    Pre-computation: M × K × K distances — once at index build time
    Per-vector distance: O(M) lookups
    """
    # centroid_distance_tables[m][i][j] = distance between 
    # centroid i and centroid j in sub-space m
    total_dist = sum(
        centroid_distance_tables[m][query_pq_code[m]][db_pq_code[m]] 
        for m in range(M)
    )
    return total_dist
```

| Method | Query Precision | DB Precision | Accuracy | Pre-computation |
|--------|----------------|-------------|----------|-----------------|
| ADC | Full (float32) | Quantized (PQ) | Higher | Per query (fast) |
| SDC | Quantized (PQ) | Quantized (PQ) | Lower | One-time (offline) |

**In practice**: ADC is almost always used because the accuracy gain is significant and the per-query table computation is negligible.

### Optimized Product Quantization (OPQ)

Standard PQ splits vectors into contiguous blocks, but correlations between dimensions may span block boundaries. **OPQ** fixes this by learning an optimal **rotation** of the vector space before quantization:

```python
def opq_encode(vector, rotation_matrix, codebooks, M, d_sub):
    """
    OPQ: rotate vector first, then apply standard PQ.
    The rotation R is learned to minimize quantization distortion.
    """
    # Step 1: Apply learned rotation
    rotated = rotation_matrix @ vector  # 768 → 768 (same dims, different basis)
    
    # Step 2: Standard PQ on rotated vector
    pq_code = []
    for m in range(M):
        sub_vec = rotated[m*d_sub : (m+1)*d_sub]
        centroid_id = nearest_centroid(sub_vec, codebooks[m])
        pq_code.append(centroid_id)
    
    return pq_code

# OPQ typically improves recall by 5-15% over standard PQ
# at the cost of ~2x longer training time
```

## IVF-PQ (The Combination)

Combine IVF's cluster pruning with PQ's compression for the ultimate scalable solution:

### How It Works

```
Training phase:
  1. k-means → nlist centroids (Voronoi partition)
  2. For each vector: compute residual = vector - nearest_centroid
  3. Train PQ codebooks on the residual vectors (not originals!)
  4. Store PQ-encoded residuals in inverted lists

Query phase:
  1. Find nearest centroids (nprobe)
  2. For each centroid: compute query_residual = query - centroid
  3. ADC search within cluster using residual PQ codes
  4. Return top-k across all probed clusters
```

**Key insight**: PQ is applied to **residual vectors** (vector - centroid), not the original vectors. Residuals are smaller in magnitude and more uniformly distributed → PQ works better on them.

```python
import faiss

# Build IVF-PQ index
nlist = 4096    # Number of Voronoi cells
M = 96          # Number of sub-vectors for PQ
nbits = 8       # Bits per sub-vector (256 centroids per sub-space)
d = 768         # Vector dimension

# Create quantizer (used to assign vectors to Voronoi cells)
quantizer = faiss.IndexFlatL2(d)

# Create IVF-PQ index
index = faiss.IndexIVFPQ(quantizer, d, nlist, M, nbits)

# Train on representative sample (at least 30 * nlist vectors)
index.train(training_vectors)  # Learns centroids + PQ codebooks

# Add all vectors
index.add(all_vectors)  # Assigns to clusters + PQ-encodes residuals

# Search
index.nprobe = 64  # Search 64 of 4096 clusters
distances, indices = index.search(query_vectors, k=10)
```

### FAISS Index Decision Matrix

| Index | Memory per Vec (768d) | Speed (1M QPS range) | Recall@10 | Best For |
|-------|----------------------|---------------------|-----------|----------|
| Flat | 3,072 B | 50 QPS | 100% | Ground truth, < 100K vecs |
| IVF-Flat | 3,072 B | 2,000 QPS | 95-99% | Medium datasets, high accuracy |
| IVF-PQ | ~96 B | 5,000+ QPS | 85-95% | Billion-scale, memory-constrained |
| IVF-PQ + refine | ~96 B + disk | 1,000 QPS | 95-98% | Best accuracy/memory tradeoff |
| HNSW (Ch 6.5) | ~3,500 B | 3,500 QPS | 98-99% | High recall, RAM available |

### Practical Configuration Guidelines

```python
# Small dataset (< 100K vectors): Flat index — don't overthink it
index = faiss.IndexFlatL2(768)

# Medium dataset (100K - 10M): IVF-Flat for accuracy, HNSW for speed
n_clusters = int(4 * math.sqrt(n_vectors))
quantizer = faiss.IndexFlatL2(768)
index = faiss.IndexIVFFlat(quantizer, 768, n_clusters)
index.nprobe = 32

# Large dataset (10M - 1B): IVF-PQ for memory efficiency
index = faiss.IndexIVFPQ(quantizer, 768, 65536, 96, 8)
index.nprobe = 128

# Billion-scale: IVF-PQ with OPQ preprocessing
index = faiss.index_factory(768, "OPQ96_768,IVF262144,PQ96")
```

## Residual Quantization and Beyond

### Residual Quantization (RQ)

Instead of splitting vectors into sub-vectors (PQ), apply quantization **iteratively** to the residual error:

```python
def residual_quantization(vector, codebooks, stages=8):
    """
    Stage 1: Quantize vector → get approximation + residual
    Stage 2: Quantize residual → get finer approximation + new residual
    ...repeat...
    
    Each stage reduces the error further.
    """
    codes = []
    residual = vector
    
    for stage in range(stages):
        centroid_id = nearest_centroid(residual, codebooks[stage])
        codes.append(centroid_id)
        residual = residual - codebooks[stage][centroid_id]  # New residual
    
    return codes  # Same storage as PQ, potentially better accuracy
```

RQ can achieve **better accuracy than PQ** at the same compression ratio because it considers the full vector structure at each stage rather than splitting into independent sub-spaces.

### ScaNN (Google)

Google's **ScaNN** (Scalable Nearest Neighbors) uses anisotropic product quantization:
- Standard PQ minimizes reconstruction error uniformly
- ScaNN weights the quantization error to prioritize the **direction** of the vector (important for cosine similarity) over the magnitude
- Result: better recall for cosine similarity searches at the same compression

## Key Takeaways

1. **Brute force** is exact but O(N×d)—usable only under ~100K vectors
2. **IVF** partitions vector space into Voronoi cells using k-means, searching only top-nprobe clusters → ~100x speedup with 1% recall loss
3. **Boundary problem**: Vectors near Voronoi cell edges may have nearest neighbors in adjacent, unprobed cells → increase nprobe to mitigate
4. **PQ** compresses vectors 32-64x by quantizing sub-vectors independently → fits billions of vectors in RAM
5. **ADC** (asymmetric distance computation) is preferred: keep query at full precision, quantize only database vectors
6. **OPQ** rotates the vector space before PQ to reduce cross-dimension correlations → 5-15% recall improvement
7. **IVF-PQ** combines cluster pruning + compression: the default choice for billion-scale deployments
8. **Residual quantization** applies iterative error refinement and can outperform PQ at same compression ratio
