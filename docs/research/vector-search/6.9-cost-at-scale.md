# Chapter 6.9: Cost of Vector Search at Scale

## Executive Summary

Vector search is expensive. This isn't a minor implementation detail — at billion-scale, the infrastructure costs of vector search can dominate a company's entire search budget. HNSW indexes require terabytes of RAM, embedding models consume GPU hours, and operational complexity multiplies as data grows. Engineers who don't understand these costs build systems that work beautifully at 1 million vectors and then face a rude awakening at 100 million. This chapter provides concrete cost analysis with real infrastructure numbers, demonstrates optimization strategies that can reduce costs by 10-30x, and gives engineers the math to budget vector search infrastructure before committing to an architecture.

---

## The Cost Components

### 1. Embedding Generation Cost

Before you can search vectors, you need to create them. Every document in your corpus must be passed through a neural network to produce its embedding. This is a one-time cost per document (plus re-embedding when documents change), but at scale it's substantial.

The cost depends on the model size: smaller models are faster and cheaper but produce lower-quality embeddings. Larger models produce better embeddings but cost more to run. API-based models are convenient but charge per token.

| Model | Dims | Speed (A100 GPU) | Cost per 1M docs | Quality (MTEB) |
|-------|------|------------------|-------------------|----------------|
| all-MiniLM-L6-v2 | 384 | ~5,000 docs/sec | ~$0.60 | 56.3 |
| all-mpnet-base-v2 | 768 | ~2,500 docs/sec | ~$1.20 | 57.8 |
| e5-large-v2 | 1024 | ~800 docs/sec | ~$3.75 | 62.2 |
| OpenAI text-embedding-3-small | 1536 | API | ~$20 (API cost) | — |
| Cohere embed-v3 | 1024 | API | ~$10 (API cost) | — |

At **1 billion documents**, these costs multiply dramatically:

- **Local (all-MiniLM on A100)**: ~$600 in GPU hours, ~55 hours of compute time. This is manageable but requires access to GPU hardware and the engineering to run batch embedding jobs.
- **Local (e5-large on A100)**: ~$3,750 in GPU hours, ~14 days of compute time on a single GPU. Parallelizing across 8 GPUs brings this down to ~2 days but costs the same in total GPU-hours.
- **API (OpenAI)**: ~$20,000 in API costs for 1B documents. No GPU hardware needed, but API rate limits mean this could take days to complete.

**Re-embedding cost**: When you upgrade to a better embedding model (which happens regularly as the field advances), you must re-embed your entire corpus. This is a full repeat of the original cost. Companies at billion-scale often delay model upgrades because the re-embedding cost is so high.

### 2. Storage Cost (Raw Vectors)

The memory required to store vectors is determined by a simple formula:

```python
memory_per_vector = dimensions × bytes_per_float

# Examples:
768_dim_float32 = 768 × 4 = 3,072 bytes = 3 KB per vector
768_dim_float16 = 768 × 2 = 1,536 bytes = 1.5 KB per vector
768_dim_int8    = 768 × 1 = 768 bytes = 0.75 KB per vector
```

These per-vector costs seem small, but they compound brutally at scale:

| Scale | float32 (768d) | float16 (768d) | int8 (768d) |
|-------|---------------|---------------|-------------|
| 1M vectors | 3 GB | 1.5 GB | 0.75 GB |
| 10M vectors | 30 GB | 15 GB | 7.5 GB |
| 100M vectors | 300 GB | 150 GB | 75 GB |
| 1B vectors | **3,000 GB (3 TB)** | 1,500 GB | 750 GB |

To put 3 TB in perspective: a high-memory cloud instance (like AWS r6g.16xlarge) has 512 GB of RAM and costs ~$3,600/month. You'd need **6 of these instances** just to hold the raw vectors — about $21,600/month before considering the index overhead.

### 3. Index Overhead

The vectors themselves are only part of the story. The index structure (HNSW graph, IVF centroid lists, etc.) adds additional memory:

**HNSW graph memory**: Each node maintains bidirectional links to its neighbors. With M=16 (default), each node stores approximately 16 × 2 × 4 = 128 bytes of graph metadata. This seems small per node but adds up:

```python
hnsw_graph_memory = N × M × 2 × sizeof(int)

# For 1B vectors, M=16:
graph_overhead = 1_000_000_000 × 16 × 2 × 4 = 128 GB

# Total HNSW memory:
total = 3_000 GB (vectors) + 128 GB (graph) = ~3,200 GB
# The graph is only ~4% overhead, but 128 GB is still a lot of RAM
```

**IVF overhead**: IVF adds centroid storage (negligible) and inverted list metadata (~8 bytes per vector for list pointers). At 1B vectors, that's ~8 GB — trivial compared to vector storage.

**The takeaway**: For HNSW at billion-scale, plan for vector storage + 20-40% overhead. The graph structure, vector storage, and working memory during search all contribute.

### 4. Infrastructure Cost (Cloud)

Here's what it actually costs to run vector search at various scales on cloud infrastructure. These numbers are based on AWS pricing (on-demand, us-east-1, as of 2024) and should be treated as illustrative — actual costs vary by provider, region, reserved pricing, and architecture choices.

| Vectors | HNSW (RAM needed) | Instance Type | Monthly Cost |
|---------|-------------------|---------------|-------------|
| 1M | ~4 GB | r6g.large (16 GB) | ~$90/mo |
| 10M | ~35 GB | r6g.xlarge (32 GB × 2) | ~$360/mo |
| 100M | ~350 GB | r6g.8xlarge (256 GB) × 2 | ~$3,600/mo |
| 1B | ~3.2 TB | r6g.16xlarge (512 GB) × 7 | ~$25,000/mo |
| 10B | ~32 TB | 70+ instances | ~$250,000/mo |

Notice the non-linear cost scaling: going from 100M to 1B vectors (10x data) costs ~7x more in infrastructure because larger instances have worse price/GB ratios and you need more of them, plus operational overhead increases.

**Managed vector database costs** (Pinecone, Weaviate Cloud, etc.) are typically 2-5x higher than self-managed cloud instances, but include operational overhead, scaling, backups, and monitoring. For teams without dedicated infrastructure engineers, the managed premium is often worth it.

## Cost Optimization Strategies

### Strategy 1: Quantization — The Single Biggest Lever

Quantization reduces the precision of each number in the vector, trading a small amount of accuracy for massive memory savings. This is almost always the first optimization to apply because it's simple, well-understood, and the recall impact is manageable.

```python
import numpy as np

# Scalar Quantization: float32 → int8
# Each dimension: 4 bytes → 1 byte (4x savings)
# How it works: map the floating-point range to 256 integer values
def scalar_quantize(vectors):
    """
    Quantize float32 vectors to int8.
    Recall loss: typically 2-5% (small and predictable).
    """
    min_vals = vectors.min(axis=0)
    max_vals = vectors.max(axis=0)
    scales = (max_vals - min_vals) / 255
    
    quantized = ((vectors - min_vals) / scales).astype(np.uint8)
    return quantized, min_vals, scales

# Product Quantization: 768-dim → 96 bytes (32x savings)
# How it works: split vector into 96 sub-vectors of 8 dims each,
# replace each sub-vector with the index of its nearest centroid (1 byte)
# Recall loss: 5-15% (significant, often needs two-phase search to compensate)

# Binary Quantization: 768-dim → 96 bytes (32x savings)  
# How it works: each dimension → 1 bit (positive → 1, negative → 0)
# Fastest distance computation (hamming distance = popcount)
# Recall loss: 10-30% (large, best used as coarse filter with full-precision re-scoring)
```

The impact on infrastructure costs is dramatic:

| Method | Compression | Recall Impact | Memory (1B vecs, 768d) | Monthly Cost |
|--------|------------|--------------|------------------------|-------------|
| float32 (baseline) | 1x | — | 3,000 GB | ~$25,000 |
| float16 | 2x | < 1% | 1,500 GB | ~$12,500 |
| Scalar int8 | 4x | 2-5% | 750 GB | ~$6,200 |
| Product Quantization | 32x | 5-15% | 96 GB | ~$800 |
| Binary Quantization | 32x | 10-30% | 96 GB | ~$800 |

Going from float32 to scalar int8 saves ~$19,000/month at billion-scale with only 2-5% recall loss. Going to PQ saves ~$24,000/month but requires careful handling of the 5-15% recall drop (typically via two-phase search).

### Strategy 2: Two-Phase Search (Oversample + Re-score)

This is the standard production pattern for using quantized indexes without sacrificing accuracy. The idea is simple: search the compressed index to find roughly the right candidates, then re-score those candidates with full-precision vectors to fix the ranking.

```python
def two_phase_search(query, compressed_index, full_vector_store, k=10, oversample=10):
    """
    Phase 1: Search compressed index (PQ/binary) → top k×oversample candidates
             Uses quantized vectors in RAM → fast, memory-efficient
    
    Phase 2: Load full-precision vectors for ONLY the candidates from SSD,
             re-compute exact distances, return top-k
             Loads ~100 vectors × 3KB = 300KB from SSD → negligible I/O
    
    Result: Near-exact recall at compressed-index memory costs
    """
    # Phase 1: Fast approximate search (compressed vectors in RAM)
    candidates = compressed_index.search(query, k=k * oversample)  # ~100 results
    
    # Phase 2: Exact re-scoring (load full vectors from SSD for candidates only)
    rescored = []
    for doc_id in candidates:
        full_vec = full_vector_store.load_from_ssd(doc_id)  # ~0.1ms per vector
        exact_score = cosine_similarity(query, full_vec)
        rescored.append((exact_score, doc_id))
    
    return sorted(rescored, reverse=True)[:k]

# Performance:
# Phase 1: ~0.3ms (PQ search over 1B compressed vectors in ~96 GB RAM)
# Phase 2: ~10ms (load 100 vectors from NVMe SSD + exact distance computation)
# Total: ~10ms — fast enough for real-time search
# Recall: ~98% (vs ~85% for PQ alone)
# Memory: ~96 GB RAM + 3 TB SSD (vs ~3.2 TB RAM for full HNSW)
```

### Strategy 3: DiskANN (SSD-Based Vector Search)

Microsoft's DiskANN stores the Vamana graph on SSD instead of RAM, keeping only compressed PQ codes in memory. This achieves a dramatic cost reduction for billion-scale deployments.

The fundamental insight is that SSD storage costs ~10x less per GB than RAM, and modern NVMe SSDs can deliver the random I/O patterns needed for graph traversal with acceptable latency (5-20ms per query vs. <1ms for in-memory HNSW).

```
HNSW (1B vectors, 768d):    ~3.2 TB RAM required   → ~$25,000/mo
DiskANN (1B vectors, 768d): ~64 GB RAM + 3 TB SSD  → ~$3,000-5,000/mo
                            → 5-8x cost reduction

Latency tradeoff:
  HNSW:    < 1ms per query (all in RAM)
  DiskANN: 5-20ms per query (SSD I/O for graph traversal)
```

For many applications, 10-20ms query latency is perfectly acceptable (web search, document retrieval, recommendation systems). The 5-8x cost savings often makes DiskANN the clear choice for billion-scale deployments where sub-millisecond latency isn't critical.

### Strategy 4: Tiered Storage

Not all vectors need to be in RAM. Production systems can implement tiered storage based on access patterns:

```
Tier 1 (RAM):     Hot vectors — recent, popular, frequently accessed
                  → HNSW, sub-millisecond latency
                  → Typically 10-20% of total corpus

Tier 2 (SSD):     Warm vectors — moderately accessed
                  → DiskANN or IVF-PQ with SSD-backed vectors
                  → 5-20ms latency, 5-10x cheaper than RAM

Tier 3 (Object):  Cold vectors — rarely accessed, archival
                  → IVF-PQ on cloud storage (S3/GCS)
                  → 100ms+ latency, 50-100x cheaper than RAM
                  → Acceptable for batch processing, analytics
```

The key is building a routing layer that directs queries to the appropriate tier. For most applications, recency is a strong signal: recent documents are queried more often, so they stay in the hot tier. Older documents migrate to colder tiers based on configurable age thresholds.

### Strategy 5: Dimensional Reduction with Matryoshka Embeddings

Lower dimensions mean less memory AND faster distance computation — a double win:

```
768-dim × 4 bytes = 3,072 bytes/vector
384-dim × 4 bytes = 1,536 bytes/vector  (2x savings)
256-dim × 4 bytes = 1,024 bytes/vector  (3x savings)
128-dim × 4 bytes =   512 bytes/vector  (6x savings)
```

**Matryoshka embeddings** (Kusupati et al., 2022) are models specifically trained to produce embeddings that are useful at any prefix length. You can take the first 256 dimensions of a 768-dim Matryoshka embedding and still get good retrieval quality — typically only 2-5% recall drop.

This is different from post-hoc dimensionality reduction (PCA), which typically loses 5-15% recall at the same reduction ratio. Matryoshka models are trained with a multi-scale loss that explicitly optimizes for quality at every prefix length.

```python
# Using Matryoshka embeddings for cost optimization
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('nomic-ai/nomic-embed-text-v1.5')  # Matryoshka-trained

# Full embedding: 768 dims, 3 KB per vector
full_vec = model.encode("affordable noise cancelling headphones")  # [768]

# Truncated: 256 dims, 1 KB per vector — 3x less memory, ~3% recall drop
truncated = full_vec[:256]  # Just take the first 256 dimensions!

# For 1B vectors:
# 768-dim: 3,000 GB RAM → ~$25,000/mo
# 256-dim: 1,000 GB RAM → ~$8,300/mo (saving $16,700/mo)
```

### Strategy 6: Sharding by Semantic Category

Instead of one monolithic index containing all vectors, partition the data into smaller, topic-specific indexes:

```
"Electronics" index: 50M vectors → fits on one r6g.8xlarge (~$1,800/mo)
"Clothing" index:    30M vectors → fits on one r6g.4xlarge (~$900/mo)  
"Books" index:       40M vectors → fits on one r6g.4xlarge (~$900/mo)
"Home" index:        20M vectors → fits on one r6g.xlarge (~$180/mo)

Total: 140M vectors across 4 shards → ~$3,780/mo
vs. one 140M-vector index: ~$5,600/mo on larger instances
```

Benefits beyond cost:
- Each index fits on a smaller, cheaper machine
- Query routing skips irrelevant shards (searching "TVs" doesn't search the "Books" index)
- Independent scaling: add replicas only for high-traffic categories
- Independent updates: re-index "Electronics" without touching "Books"

The tradeoff is increased operational complexity: you need a routing layer, health monitoring per shard, and a strategy for cross-category queries.

## The Total Cost of Ownership Formula

When budgeting for vector search, consider all cost components:

```python
monthly_cost = (
    embedding_cost +          # GPU/API for encoding new/changed docs
    infrastructure_cost +     # Cloud instances for index hosting
    operational_cost          # Engineering time, monitoring, reindexing
)

embedding_cost = (
    new_docs_per_month / embedding_speed_per_second * 3600 
    * gpu_hourly_rate
)

infrastructure_cost = (
    num_instances * instance_hourly_rate * 720  # 720 hours/month
    + storage_gb * storage_rate_per_gb_month
)

operational_cost = (
    engineering_hours_per_month * hourly_rate
    + monitoring_tools
    + periodic_reindexing_compute
)
```

### Worked Example: 100M Product Catalog

Let's walk through a realistic budget for an e-commerce company with 100M products:

```
Embedding (one-time + monthly updates):
  100M docs × all-mpnet-base (2500 docs/sec)
  = 40,000 seconds = 11 hours on 1× A100
  = ~$33 per full re-embedding
  Monthly: ~5M new/changed products = ~$1.65 + 1 GPU hour margin = ~$5/mo

Infrastructure:
  HNSW with scalar quantization (int8):
  100M × 768 bytes = 75 GB vectors + 15 GB graph = ~90 GB index
  2× r6g.4xlarge (128 GB RAM each, for redundancy) = ~$1,800/mo
  + 500 GB SSD for full vectors (two-phase re-scoring) = ~$50/mo
  Infrastructure total: ~$1,850/mo

Operational:
  ~10 hours/month engineering × $100/hr = $1,000/mo
  Monitoring (Datadog/similar) = ~$200/mo
  Quarterly full re-index (4× $33 / 12) = ~$11/mo
  Operational total: ~$1,211/mo

Grand total: ~$3,066/month for 100M vectors with good recall

Without quantization (float32 HNSW):
  300 GB vectors + 40 GB graph = ~340 GB
  2× r6g.8xlarge (256 GB) = ~$3,600/mo
  Grand total: ~$4,811/month

Savings from quantization: ~$1,750/month = ~36% reduction
```

## Cost Comparison Across Architectures

| Architecture | Memory (100M, 768d) | Monthly Infra Cost | Query Latency | Recall@10 |
|-------------|--------------------|--------------------|---------------|-----------|
| HNSW float32 | 340 GB | ~$3,600 | < 1ms | 98% |
| HNSW int8 | 90 GB | ~$1,800 | < 1ms | 95% |
| HNSW int8 + rescore | 90 GB + SSD | ~$1,850 | ~5ms | 97% |
| IVF-PQ | 12 GB | ~$180 | ~2ms | 88% |
| IVF-PQ + rescore | 12 GB + SSD | ~$230 | ~10ms | 96% |
| DiskANN | 8 GB + SSD | ~$200 | ~15ms | 95% |

The right architecture depends on your latency requirements. If you can tolerate 10-15ms latency (most applications can), DiskANN or IVF-PQ with re-scoring gives you 95%+ recall at a fraction of HNSW's cost.

## Key Takeaways

1. **Memory is the dominant cost**: 1B vectors × 768-dim float32 = 3 TB of RAM, costing ~$25K/month on cloud infrastructure. Understanding this math is essential before committing to an architecture.
2. **Quantization is the single biggest cost lever**: Scalar quantization (4x) saves ~36% of infrastructure cost with only 2-5% recall loss. Product quantization (32x) is even more dramatic but requires two-phase search to maintain quality.
3. **Two-phase search is the production standard**: Search compressed index in RAM → re-score top candidates with full-precision vectors from SSD. This gives near-exact recall at compressed-index memory costs.
4. **DiskANN** offers 5-8x cost reduction for billion-scale by moving the graph to SSD, trading <1ms latency for 5-20ms.
5. **Matryoshka embeddings** enable dimensional reduction (768→256) with only 2-5% recall loss, providing an additional 3x memory savings that compounds with quantization.
6. **Budget for operational costs**: Re-embedding (model upgrades, new data), re-indexing (HNSW graph rebuild), and engineering time are ongoing costs that are easy to underestimate.
7. **Managed vs. self-managed**: Managed vector databases (Pinecone, Weaviate Cloud) cost 2-5x more but eliminate operational overhead. Worth it for teams without dedicated infrastructure engineers.
