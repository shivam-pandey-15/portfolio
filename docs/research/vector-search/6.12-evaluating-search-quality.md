# Chapter 6.12: Evaluating Search Quality

## Executive Summary

Building a search system without measuring its quality is like driving blindfolded. You might be going in the right direction, but you have no way to know — and no way to detect when you're drifting off course. Yet many teams deploy semantic search systems with no formal evaluation, relying on subjective impressions ("the results look good") or anecdotal feedback ("a user complained"). This chapter covers the standard evaluation metrics (Recall, Precision, MRR, nDCG, MAP), explains how to build evaluation datasets, and describes the offline/online evaluation pipeline that production search teams use to continuously measure and improve search quality.

---

## Why Evaluation Is Hard for Semantic Search

Traditional keyword search evaluation is relatively straightforward: you can inspect which terms matched and understand why a document was returned. With semantic search, the "why" is hidden inside a 768-dimensional vector space. A document was returned because its embedding was geometrically close to the query embedding — but this provides no interpretable explanation.

This opacity makes evaluation both more important and more difficult:

- **No obvious failures**: Keyword search returns zero results when it fails (no matching terms). Semantic search always returns something — but those results might be confidently wrong. A search for "jaguar speed" might return results about the car instead of the animal, with high similarity scores.
- **Subjective relevance**: Is a document about "budget flights to Europe" relevant to the query "cheap airfare"? It's clearly similar, but if the user wanted domestic flights, it's wrong. Relevance depends on user intent, which is often ambiguous.
- **Graded relevance**: Documents aren't just "relevant" or "not relevant." A product spec sheet is somewhat relevant to "what laptop should I buy?", but a detailed comparison review is much more relevant. Evaluation metrics must capture this gradation.

## Core Metrics

### Recall@K

Recall@K measures how many of the truly relevant documents appear in the top K results. This is the most important metric for retrieval systems because a document that isn't retrieved can never be shown to the user — no amount of downstream reranking or processing can recover it.

```python
def recall_at_k(retrieved, relevant, k):
    """
    What fraction of relevant documents did we find in the top K?
    
    retrieved: list of doc IDs returned by the system (in rank order)
    relevant: set of doc IDs that are actually relevant (ground truth)
    k: cutoff position
    
    Returns: float between 0.0 and 1.0
    """
    retrieved_at_k = set(retrieved[:k])
    relevant_found = retrieved_at_k & relevant
    return len(relevant_found) / len(relevant) if relevant else 0.0

# Example:
# Relevant docs: {A, B, C, D, E}
# Retrieved top-10: [A, X, B, Y, Z, C, W, V, U, T]
# recall@10 = |{A, B, C}| / |{A, B, C, D, E}| = 3/5 = 0.60
# recall@5  = |{A, B}| / |{A, B, C, D, E}| = 2/5 = 0.40
```

**When to use**: Always. Recall@K is the foundation metric for retrieval. If recall is low, nothing else matters — the right documents aren't making it into the candidate set. For multi-stage pipelines (retrieve-then-rerank), recall@K of the retrieval stage determines the ceiling for the entire system.

### Precision@K

Precision@K measures how many of the top K returned results are actually relevant. While recall asks "did we find everything relevant?", precision asks "is what we found actually useful?"

```python
def precision_at_k(retrieved, relevant, k):
    """
    What fraction of the top K results are relevant?
    
    Returns: float between 0.0 and 1.0
    """
    retrieved_at_k = set(retrieved[:k])
    relevant_found = retrieved_at_k & relevant
    return len(relevant_found) / k

# Example:
# Relevant docs: {A, B, C, D, E}
# Retrieved top-10: [A, X, B, Y, Z, C, W, V, U, T]
# precision@10 = |{A, B, C}| / 10 = 0.30
# precision@5  = |{A, B}| / 5 = 0.40
# precision@3  = |{A, B}| / 3 = 0.67 (A at rank 1, X misses, B at rank 3)
```

**When to use**: When returned results are directly shown to users (not filtered by a reranker). Low precision means users see many irrelevant results, which degrades trust and engagement.

### Mean Reciprocal Rank (MRR)

MRR measures how quickly the system returns the **first** relevant result. It's calculated as the average of 1/rank of the first relevant result across all queries.

```python
def mrr(queries_results, queries_relevant):
    """
    MRR: average of 1/rank_of_first_relevant across all queries.
    
    If the first relevant result is at rank 1: score = 1.0
    If the first relevant result is at rank 3: score = 0.33
    If no relevant result is found: score = 0.0
    """
    reciprocal_ranks = []
    
    for retrieved, relevant in zip(queries_results, queries_relevant):
        rr = 0.0
        for rank, doc_id in enumerate(retrieved, start=1):
            if doc_id in relevant:
                rr = 1.0 / rank
                break
        reciprocal_ranks.append(rr)
    
    return sum(reciprocal_ranks) / len(reciprocal_ranks)

# Example across 3 queries:
# Query 1: first relevant at rank 1 → 1/1 = 1.0
# Query 2: first relevant at rank 3 → 1/3 = 0.33
# Query 3: first relevant at rank 5 → 1/5 = 0.20
# MRR = (1.0 + 0.33 + 0.20) / 3 = 0.51
```

**When to use**: When users primarily care about the **first good result** — question-answering, navigation queries ("find the settings page"), fact lookup. MRR is less useful when users browse multiple results (e-commerce, document exploration).

### nDCG@K (Normalized Discounted Cumulative Gain)

nDCG is the gold standard metric for ranking quality. Unlike recall and precision (which treat relevance as binary — relevant or not), nDCG handles **graded relevance** (e.g., 0=irrelevant, 1=somewhat relevant, 2=highly relevant) and accounts for position (higher-ranked results matter more).

```python
import math

def dcg_at_k(relevance_scores, k):
    """
    DCG: sum of relevance scores, discounted by log of position.
    Higher-ranked results contribute more to the score.
    
    relevance_scores: list of graded relevance values for each result
                      e.g., [3, 2, 0, 1, 0] (in rank order)
    """
    dcg = 0.0
    for i in range(min(k, len(relevance_scores))):
        dcg += relevance_scores[i] / math.log2(i + 2)  # log2(rank+1), rank=1→log2(2)
    return dcg

def ndcg_at_k(retrieved_relevance, ideal_relevance, k):
    """
    nDCG: normalize DCG by the ideal (perfect) ranking's DCG.
    
    retrieved_relevance: relevance of results in the order returned
    ideal_relevance: relevance scores sorted in best possible order
    
    Returns: float between 0.0 and 1.0
      1.0 = perfect ranking (best possible ordering)
      0.0 = no relevant results at all
    """
    actual_dcg = dcg_at_k(retrieved_relevance, k)
    ideal_dcg = dcg_at_k(sorted(ideal_relevance, reverse=True), k)
    
    if ideal_dcg == 0:
        return 0.0
    return actual_dcg / ideal_dcg

# Example:
# System returns: [3, 2, 0, 1, 0] (a 3-relevance doc first, then 2, then 0, etc.)
# Ideal ordering: [3, 2, 1, 0, 0] (all relevant docs sorted by relevance)
# nDCG@5 = DCG([3,2,0,1,0]) / DCG([3,2,1,0,0])
#        = (3/1 + 2/1.58 + 0/2 + 1/2.32 + 0/2.58) / (3/1 + 2/1.58 + 1/2 + 0/2.32 + 0/2.58)
#        = 4.70 / 4.77 = 0.985 (nearly perfect ranking!)
```

**When to use**: Whenever you have graded relevance judgments. nDCG is the most informative single metric for search quality because it captures both what was returned (accuracy) and where it was ranked (ordering). It's the primary metric used by academic IR competitions and major search companies.

### MAP (Mean Average Precision)

MAP summarizes precision at every point where a relevant document is found:

```python
def average_precision(retrieved, relevant):
    """
    Compute average of precision values at each relevant hit position.
    
    Captures both precision AND recall in a single number:
    - Penalized when relevant docs are ranked low (poor precision at that point)
    - Penalized when relevant docs are missed entirely (they never contribute)
    """
    relevant_count = 0
    precision_sum = 0.0
    
    for rank, doc_id in enumerate(retrieved, start=1):
        if doc_id in relevant:
            relevant_count += 1
            precision_sum += relevant_count / rank
    
    return precision_sum / len(relevant) if relevant else 0.0

def mean_average_precision(all_queries_results, all_queries_relevant):
    """MAP: average AP across all queries."""
    aps = [average_precision(ret, rel) for ret, rel in 
           zip(all_queries_results, all_queries_relevant)]
    return sum(aps) / len(aps)

# Example:
# Relevant: {A, B, C}
# Retrieved: [A, X, B, Y, C]
# AP = (1/1 + 2/3 + 3/5) / 3 = (1.0 + 0.67 + 0.60) / 3 = 0.76
```

**When to use**: MAP is the standard metric for binary relevance (relevant/not relevant). It's less common for semantic search evaluation (where nDCG with graded relevance is preferred) but still widely used in IR research.

## Choosing the Right Metric

| Metric | Relevance Type | Position-Aware? | Best For |
|--------|---------------|-----------------|----------|
| Recall@K | Binary | No (just top K) | Retrieval stage evaluation |
| Precision@K | Binary | No (just top K) | User-facing result quality |
| MRR | Binary | Yes (first hit) | Question answering, navigation |
| nDCG@K | Graded | Yes (log discount) | Overall ranking quality (gold standard) |
| MAP | Binary | Yes (avg precision) | Academic benchmarks, binary relevance |

**Recommendation**: Use **Recall@K for retrieval** (did we find the right candidates?) and **nDCG@K for ranking** (are they in the right order?). If you can only track one metric, use nDCG@10.

## Building an Evaluation Dataset

The hardest part of search evaluation isn't computing metrics — it's getting **ground truth relevance judgments**. You need a set of (query, document, relevance_score) triples that represent what a good search system should return.

### Approach 1: Human Annotation

The gold standard. Human annotators judge the relevance of documents for a set of queries.

```python
# Annotation guidelines for graded relevance:
RELEVANCE_SCALE = {
    0: "Not relevant — document has nothing to do with the query",
    1: "Marginally relevant — mentions the topic but doesn't answer the query",
    2: "Somewhat relevant — partially answers the query, missing key details",
    3: "Highly relevant — directly and completely answers the query"
}

# Minimum viable evaluation set:
# - 50-100 representative queries (covering head, torso, and tail)
# - 10-20 judged documents per query
# - 2+ annotators per judgment (for inter-annotator agreement)
# Total: 1000-4000 judgments (achievable in 1-2 days of annotation)
```

**Practical tips**:
- Include queries from your actual query logs, not just made-up test queries
- Include "hard" queries (ambiguous, short, long-tail) — these are where search quality matters most
- Measure inter-annotator agreement (Cohen's kappa > 0.7 is good) — if annotators disagree, the evaluation is unreliable
- Re-annotate periodically as your corpus changes

### Approach 2: Click-Through Data

Use search logs to infer relevance from user behavior:

```python
def infer_relevance_from_clicks(query, results, clicks, dwell_times):
    """
    Infer relevance from user behavior:
    - Clicked + long dwell (>30s) = highly relevant (3)
    - Clicked + short dwell (<5s) = marginally relevant (1)
    - Not clicked but viewed = not relevant (0)
    - Skipped (results above it were clicked) = explicitly not relevant (0)
    
    Caveats:
    - Position bias: users click higher results regardless of relevance
    - Presentation bias: result snippets affect click probability
    - Not clicking ≠ not relevant (user might have found their answer in the snippet)
    """
    relevance = {}
    for result in results:
        if result in clicks:
            dwell = dwell_times[result]
            relevance[result] = 3 if dwell > 30 else 1
        else:
            relevance[result] = 0
    return relevance
```

**Advantage**: Scales to millions of queries automatically.
**Disadvantage**: Biased by position (rank 1 gets more clicks regardless), noisy, and can't evaluate unchosen results.

### Approach 3: LLM-as-Judge

Use a large language model to judge relevance:

```python
def llm_judge(query, document, model="gpt-4"):
    """
    Use LLM to assess document relevance to query.
    
    Faster and cheaper than human annotation.
    Quality: ~80-90% agreement with human judgments.
    Cost: ~$0.01-0.05 per judgment.
    """
    prompt = f"""Rate the relevance of this document to the query.

Query: {query}
Document: {document}

Relevance scale:
0 = Not relevant at all
1 = Marginally relevant (mentions topic but doesn't answer)
2 = Somewhat relevant (partially answers)
3 = Highly relevant (directly and completely answers)

Return ONLY the number (0, 1, 2, or 3)."""
    
    return int(model.generate(prompt))
```

**Advantage**: Cheap, fast, consistent, no annotator fatigue. Can generate thousands of judgments in minutes.
**Disadvantage**: LLMs have their own biases, may disagree with human preferences, and can be fooled by superficially relevant but actually wrong documents.

## Offline vs Online Evaluation

### Offline Evaluation

**Definition**: Measured on a static evaluation dataset before deploying changes. This is your test suite for search quality.

```python
def offline_evaluation(retriever, eval_dataset):
    """
    Run offline evaluation: measure metrics on fixed evaluation queries.
    
    eval_dataset: list of (query, relevant_docs, relevance_scores) triples
    """
    results = {}
    
    for query, relevant_docs, rel_scores in eval_dataset:
        retrieved = retriever.search(query, k=100)
        
        results[query] = {
            'recall@10': recall_at_k(retrieved, relevant_docs, 10),
            'recall@100': recall_at_k(retrieved, relevant_docs, 100),
            'ndcg@10': ndcg_at_k(
                [rel_scores.get(d, 0) for d in retrieved[:10]],
                list(rel_scores.values()),
                10
            ),
            'mrr': reciprocal_rank(retrieved, relevant_docs),
        }
    
    # Aggregate
    for metric in ['recall@10', 'recall@100', 'ndcg@10', 'mrr']:
        avg = mean(r[metric] for r in results.values())
        print(f"{metric}: {avg:.4f}")
    
    return results
```

**When to use**: Before every search system change — new embedding model, parameter tuning, index rebuild, chunking strategy change. If offline metrics degrade, don't deploy.

### Online Evaluation

**Definition**: Measured on live traffic after deploying changes. This captures real-world user behavior that offline evaluation can't.

Key online metrics:

| Metric | What It Measures | Signal |
|--------|-----------------|--------|
| Click-through rate (CTR) | % of queries where user clicked a result | Higher = results look relevant |
| Zero-result rate | % of queries returning no results | Lower = better coverage |
| Abandonment rate | % of queries where user left without clicking | Lower = results are useful |
| Mean dwell time | How long users spend on clicked results | Longer = content matched intent |
| Reformulation rate | % of queries where user immediately searches again | Lower = first results were good |
| SERP engagement | Scrolling depth, clicks per query, time on SERP | Higher = users finding value |

### The Offline → Online Pipeline

```
1. Develop change (new model, new params, new chunking)
                ↓
2. Run offline evaluation against eval dataset
   - If metrics improve → proceed
   - If metrics degrade → iterate, don't deploy
                ↓
3. Shadow deployment: run new system alongside old, compare results
   - Log both sets of results, don't show new results to users
   - If offline metrics hold on live queries → proceed
                ↓
4. A/B test: show new results to 5-10% of users
   - Monitor online metrics (CTR, abandonment, dwell time)
   - Run for 1-2 weeks for statistical significance
                ↓
5. If online metrics improve → full rollout
   If degradation → rollback, investigate
```

This pipeline prevents the common failure mode where an engineer changes the embedding model, sees better results on a handful of test queries, deploys to production, and discovers a week later that precision dropped 15% on a query type they didn't test.

## Key Takeaways

1. **Recall@K** is the foundation metric for retrieval — if relevant documents aren't in the candidate set, nothing downstream can fix it
2. **nDCG@K** is the gold standard for ranking quality — handles graded relevance and position-aware weighting. If you track one metric, make it nDCG@10
3. **Build an evaluation dataset** of at least 50-100 queries with human-judged relevance — this is the most impactful investment you can make in search quality
4. **LLM-as-judge** offers a scalable alternative to human annotation, achieving ~80-90% agreement at a fraction of the cost
5. **Offline evaluation before every change**: Run the evaluation pipeline before deploying any search system modification. No exceptions.
6. **Online evaluation validates real impact**: Click-through rate, abandonment rate, and dwell time capture user satisfaction that offline metrics can't
7. **The full pipeline** is: offline eval → shadow deploy → A/B test → full rollout. Skipping steps causes silent quality degradation.
