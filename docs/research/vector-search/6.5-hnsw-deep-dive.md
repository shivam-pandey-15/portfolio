# Chapter 6.5: HNSW Deep Dive

## Executive Summary

Hierarchical Navigable Small World (HNSW) is the **dominant graph-based algorithm** for approximate nearest neighbor search. It achieves O(log N) search complexity by building a multi-layer graph where upper layers provide long-range "highway" connections and lower layers provide fine-grained local connections. HNSW is the default vector index in Lucene (and therefore Elasticsearch/Solr), Qdrant, Weaviate, and pgvector.

---

## Foundational Concepts

### Navigable Small World (NSW) Graphs

Before HNSW, there was NSW:
- Build a graph where each node connects to its approximate nearest neighbors
- Search via **greedy traversal**: start from any node, always move to the neighbor closest to the query
- The "small world" property ensures short paths exist between any two nodes

**Problem with flat NSW**: Search complexity is O(N^(1/2)) — polylogarithmic but not logarithmic. Long-range connections are present but mixed with short-range ones, making routing inefficient.

### Skip Lists Inspiration

HNSW's breakthrough was layering the graph like a **probability skip list**:

```
Layer 3:  [A] ────────────────────── [D]              (express highway)
Layer 2:  [A] ──── [C] ──────────── [D] ── [F]       (regional roads)
Layer 1:  [A] ─ [B] ─ [C] ─ [D] ─ [E] ─ [F] ─ [G]  (local streets)
Layer 0:  [A] [B] [C] [D] [E] [F] [G] [H] [I] [J]   (all nodes)
```

- Layer 0: ALL nodes with dense local connections
- Higher layers: exponentially fewer nodes with longer-range connections
- Search starts at top → coarse zoom → fine zoom at bottom

## Construction Algorithm

### Layer Assignment

Each new vector is assigned a maximum layer using a geometric distribution:

```python
def assign_layer(m_L=1/ln(M)):
    """
    Randomly assign a max layer to a new node.
    Most nodes stay at layer 0. Few reach top layers.
    
    m_L controls layer density. Default: 1/ln(M)
    """
    return floor(-ln(random()) * m_L)

# Probability distribution (M=16, m_L ≈ 0.36):
# Layer 0: ~63% of nodes
# Layer 1: ~23% of nodes
# Layer 2: ~9% of nodes
# Layer 3: ~3% of nodes
# Layer 4: ~1% of nodes
```

### Insertion Algorithm

```python
def insert(new_node, graph, M, ef_construction):
    """
    Insert a new vector into the HNSW graph.
    
    Args:
        new_node: vector to insert
        graph: existing HNSW graph
        M: max connections per node per layer
        ef_construction: search width during construction (quality knob)
    """
    # 1. Assign random layer
    new_layer = assign_layer()
    
    # 2. Start from entry point at top layer
    entry_point = graph.entry_point
    current = entry_point
    
    # 3. Greedy search from top to new_layer+1 (just find nearest)
    for layer in range(graph.max_layer, new_layer + 1, -1):
        current = greedy_search(new_node, current, layer, ef=1)
    
    # 4. For each layer from new_layer down to 0:
    for layer in range(min(new_layer, graph.max_layer), -1, -1):
        # Search with beam width = ef_construction
        candidates = beam_search(new_node, current, layer, ef=ef_construction)
        
        # Select M neighbors (using heuristic for diversity)
        neighbors = select_neighbors_heuristic(new_node, candidates, M)
        
        # Add bidirectional edges
        for neighbor in neighbors:
            graph.add_edge(new_node, neighbor, layer)
            graph.add_edge(neighbor, new_node, layer)
            
            # Prune neighbor if it exceeds max connections
            if len(graph.neighbors(neighbor, layer)) > M_max:
                graph.prune(neighbor, layer, M_max)
        
        current = candidates[0]  # Best candidate for next layer
    
    # 5. Update entry point if new node reaches higher layer
    if new_layer > graph.max_layer:
        graph.entry_point = new_node
```

### Neighbor Selection Heuristic

Simple nearest-neighbor selection creates **clusters** — nodes all connect to the same popular neighbors. HNSW uses a **diversity heuristic**:

```python
def select_neighbors_heuristic(query, candidates, M):
    """
    Select M neighbors that are diverse (not all in same direction).
    Prefers neighbors that provide SHORT PATHS through the graph.
    """
    result = []
    working_set = sorted(candidates, key=lambda c: distance(query, c))
    
    for candidate in working_set:
        if len(result) >= M:
            break
        
        # Add candidate if it's closer to query than to any existing neighbor
        # This ensures diversity: neighbors aren't all clustered together
        if all(distance(query, candidate) < distance(candidate, existing) 
               for existing in result):
            result.append(candidate)
    
    return result
```

## Search Algorithm

```python
def search(query, graph, K, ef_search):
    """
    Search HNSW graph for K nearest neighbors.
    
    Args:
        query: query vector
        graph: HNSW graph
        K: number of results
        ef_search: beam width (accuracy/speed knob)
    
    Returns:
        K approximate nearest neighbors
    """
    # 1. Start at entry point, greedy search from top to layer 1
    current = graph.entry_point
    for layer in range(graph.max_layer, 0, -1):
        current = greedy_search(query, current, layer, ef=1)
    
    # 2. At layer 0, do beam search with ef_search width
    candidates = beam_search(query, current, layer=0, ef=ef_search)
    
    # 3. Return top K from candidates
    return sorted(candidates, key=lambda c: distance(query, c))[:K]
```

### Search Trace Example

```
Query: find neighbors of vector Q = [0.5, 0.3, ...]

Layer 3: Entry point A (dist=0.8)
         → A's neighbor D (dist=0.4) ← closer! Move to D
         → No improvement... descend with D

Layer 2: Start at D (dist=0.4)
         → D's neighbor C (dist=0.35) ← closer!
         → C's neighbor F (dist=0.5) ← worse
         → Descend with C

Layer 1: Start at C (dist=0.35)
         → Beam search with ef=64
         → Found: C(0.35), E(0.12), G(0.08), H(0.15), ...
         → Descend with G

Layer 0: Start at G (dist=0.08)
         → Beam search with ef=200
         → Explore G's neighbors, their neighbors, ...
         → Results: [G(0.08), I(0.05), J(0.06), K(0.09), ...]
```

## Complexity Analysis

| Operation | Complexity | Notes |
|-----------|-----------|-------|
| Construction (per element) | O(log N) | Search + edge creation per layer |
| Construction (total) | O(N log N) | Insert N elements |
| Search | O(log N) | Layers = log N, each layer is O(1) avg hops |
| Memory (vectors) | O(N × d) | Store all vectors |
| Memory (graph) | O(N × M) | M edges per node, ~4 bytes per edge |
| Memory (total) | O(N × (d + M)) | Vectors + graph overhead |

### Memory Formula

```
Total memory = N × (d × sizeof(float) + M × 2 × sizeof(int))

Example: 10M vectors, 768-dim, M=16
  Vectors: 10M × 768 × 4 bytes = 30.7 GB
  Graph:   10M × 16 × 2 × 4 bytes = 1.28 GB
  Total:   ~32 GB

Example: 1B vectors, 768-dim, M=16
  Vectors: 1B × 768 × 4 bytes = 3,072 GB (3 TB!)
  Graph:   1B × 16 × 2 × 4 bytes = 128 GB
  Total:   ~3.2 TB  ← Requires massive RAM or disk-backed approach
```

## Key Parameters

| Parameter | Typical Range | Effect | Memory Impact |
|-----------|--------------|--------|---------------|
| `M` | 8-64 (default 16) | Max connections per node per layer. Higher = better recall, more memory, slower build | +8 bytes/node per M increase |
| `ef_construction` | 64-512 (default 200) | Build-time beam width. Higher = better graph quality, slower build | No memory impact (build-time only) |
| `ef_search` | 32-512 | Query-time beam width. Higher = better recall, higher latency | No memory impact (query-time only) |
| `M_max` | 2×M | Max connections at layer 0 (typically double M for denser base layer) | Affects layer 0 density |

### Parameter Interaction Effects

Doubling M from 16 → 32:
- Memory: ~2x increase in graph overhead
- Index build time: ~4x increase (more neighbors to evaluate)
- Search recall@10: 95% → 98% (significant gain)
- Search latency: 1.2x increase (more edges to traverse)

Doubling ef_construction from 200 → 400:
- Memory: No change
- Index build time: ~2x increase
- Search recall@10: ~1-2% improvement (better graph quality)
- Search latency: No change (graph quality, not search-time parameter)

### Tuning Guidelines

```
For high recall (>99%):
  M = 32-48, ef_construction = 400, ef_search = 200-500

For balanced (95-98% recall):
  M = 16, ef_construction = 200, ef_search = 64-128

For speed-optimized (~90% recall):
  M = 8-12, ef_construction = 100, ef_search = 32-64
```

## Construction Benchmarks

### Build Time at Scale

| Dataset Size | Dimensions | M | ef_construction | Build Time | Hardware |
|-------------|-----------|---|----------------|-----------|----------|
| 1M | 768 | 16 | 200 | ~2 min | 32-core CPU |
| 10M | 768 | 16 | 200 | ~25 min | 32-core CPU |
| 100M | 768 | 16 | 200 | ~1.5 hours | 48-core Xeon |
| 1B | 768 | 16 | 200 | ~20 hours | 48-core Xeon |

**Key observations**:
- Construction is **embarrassingly parallel** — multiple insertions can proceed simultaneously
- Build time scales **slightly super-linearly** with N (more hops needed as graph grows)
- Higher M dramatically increases build time: M=32 takes ~4x longer than M=16
- Memory during construction is ~1.5x the final index size (working buffers)

### What Can Go Wrong at Scale

```
"What works at 10M vectors can collapse catastrophically at 100M+"

Common issues:
1. Memory explosion: HNSW graph + vectors exceeds RAM → thrashing to disk
   → Solution: Estimate memory before building, use quantization

2. Construction stalls: Late insertions take much longer as graph is dense
   → Solution: Parallelize construction, batch insertions

3. Quality degradation: If ef_construction is too low for dataset size
   → Solution: Scale ef_construction with sqrt(N)

4. Merge storms (Lucene): Segment merges trigger full graph rebuilds
   → Solution: Use larger initial segments, merge policy tuning
```

## HNSW vs DiskANN: The Memory/Latency Tradeoff

DiskANN (Microsoft, 2019) uses the **Vamana graph algorithm** stored on SSD instead of RAM:

### Architecture Comparison

| Aspect | HNSW | DiskANN |
|--------|------|---------|
| Graph storage | RAM (all layers) | SSD (graph + vectors), RAM (PQ codes only) |
| Memory per vector | ~4 KB (768d + graph) | ~200 bytes (PQ codes + metadata) |
| Query latency | < 1ms | 5-20ms (SSD I/O bound) |
| Recall@10 | 98-99% | 95-98% |
| Build time | O(N log N) | O(N log N), but single-pass |
| Updates | Online (insert/delete) | Batch (rebuild for changes) |
| Architecture | Multi-layer hierarchical | Single-layer flat graph |

### Memory Comparison at Scale

| Vectors | HNSW (RAM) | DiskANN (RAM + SSD) | Cost Ratio |
|---------|-----------|---------------------|------------|
| 10M | ~40 GB RAM | ~2 GB RAM + 40 GB SSD | HNSW: 8x more expensive |
| 100M | ~400 GB RAM | ~20 GB RAM + 400 GB SSD | HNSW: 10x more expensive |
| 1B | ~3.2 TB RAM | ~64 GB RAM + 3.2 TB SSD | **HNSW: 15-50x more expensive** |

### When to Use Each

```
Use HNSW when:
  - Dataset < 100M vectors
  - Sub-millisecond latency is critical
  - Frequent updates to the index
  - Budget allows sufficient RAM

Use DiskANN when:
  - Dataset > 100M vectors
  - 5-20ms latency is acceptable
  - Index is relatively static (batch updates OK)
  - Memory budget is constrained
  - Cost optimization is priority
```

### FreshDiskANN

The original DiskANN required batch rebuilds. **FreshDiskANN** adds online updates:
- Maintains a small in-memory "streaming" index for recent inserts
- Periodically merges streaming index into the on-disk Vamana graph
- Handles deletes via tombstoning + periodic compaction
- Trade-off: slightly higher latency during merge windows

## HNSW in Lucene / Elasticsearch

Lucene 9+ uses HNSW for kNN search:

```json
// Elasticsearch kNN query
{
  "knn": {
    "field": "my_vector",
    "query_vector": [0.1, 0.2, ...],
    "k": 10,
    "num_candidates": 100
  }
}
```

### Lucene's Segment Architecture with HNSW

Unlike standalone HNSW (single graph), Lucene maintains **per-segment HNSW graphs**:

```
Segment 1:  HNSW graph (500K vectors) ── created at time T1
Segment 2:  HNSW graph (300K vectors) ── created at time T2
Segment 3:  HNSW graph (200K vectors) ── created at time T3

Query → search all 3 graphs → merge results → return top-K

After segment merge:
Merged Segment: HNSW graph (1M vectors) ── REBUILT from scratch
```

### The Merge Problem

HNSW graph merging is Lucene's biggest challenge with vector search:

1. **Graph rebuild on merge**: When segments merge, the HNSW graph is essentially **rebuilt from scratch** — the hierarchical structure from individual segments cannot be efficiently combined
2. **Memory spikes**: 9M vectors with M=16 can allocate over 2.5 GB of heap during merge
3. **CPU cost**: Merge requires re-inserting all vectors into a new graph = O(N log N)
4. **Impact on indexing**: Long merges block new segment creation, causing indexing latency spikes

### Lucene Mitigations

```
1. Specialized HNSW graph merging (Lucene 9.6+):
   - Predict memory usage before merge → reject if too large
   - Process HNSW levels one at a time → free lower-level memory
   - Incrementally process neighbors → prevent heap spikes

2. Merge policy tuning:
   - Use larger initial segments (fewer merges)
   - TieredMergePolicy: limit max merge segment size
   - Separate vector fields into dedicated segments

3. Off-heap storage:
   - Vectors stored via mmap, not on Java heap
   - Graph navigation structures in off-heap memory
   - Reduces GC pressure during merges
```

### Lucene Configuration

```json
// Elasticsearch vector field mapping
{
  "mappings": {
    "properties": {
      "my_vector": {
        "type": "dense_vector",
        "dims": 768,

```

| Elasticsearch Setting | Default | Production Recommendation |
|----------------------|---------|--------------------------|
| `m` | 16 | 16-32 (higher for better recall) |
| `ef_construction` | 100 | 200-400 (worth the build cost) |
| `num_candidates` (ef_search) | varies | 100-200 for top-10 queries |
| `similarity` | cosine | cosine for text, l2 for images |

## Key Takeaways

1. **HNSW** achieves O(log N) search by layering a navigable small world graph like a probability skip list
2. **Layer assignment** is probabilistic via geometric distribution: ~63% of nodes only in layer 0, ~1% reach layer 4+
3. **Construction** is O(N log N): each insertion involves a beam search + diverse neighbor selection with pruning heuristic
4. **ef_search** is the primary accuracy/speed knob at query time; M and ef_construction affect build quality
5. **Memory overhead**: ~20-40% on top of vector storage for the graph structure. 1B vectors × 768-dim = ~3.2 TB
6. **DiskANN alternative**: 15-50x less memory than HNSW by storing graph on SSD, but 5-20ms latency vs <1ms
7. **Lucene integration**: Per-segment HNSW graphs that must be rebuilt from scratch during segment merges — the biggest operational challenge
8. **Construction at scale**: 100M vectors takes ~1.5 hours; can collapse if memory is insufficient. Parallelize construction and monitor memory
