# Chapter 6.6: Latency vs Recall Tradeoffs

## Executive Summary

Every vector search system faces the same fundamental tradeoff: **better accuracy costs more time and memory**. This isn't a limitation that can be engineered away — it's an inherent property of approximate nearest neighbor search. The art of production vector search lies in understanding this tradeoff deeply, knowing which knobs to turn, and finding the operating point that balances your application's accuracy requirements against its latency and cost constraints. This chapter quantifies these relationships with concrete benchmarks and provides a decision framework.

---

## Defining the Metrics

### Recall@K — The Key Accuracy Metric

Recall@K is the fraction of true K-nearest neighbors returned by the approximate algorithm compared to an exact brute-force search:

```python
def recall_at_k(approx_results, exact_results, k):
    """
    If exact brute-force returns [A, B, C, D, E, F, G, H, I, J] as top-10,
    and ANN returns [A, B, C, D, E, F, G, X, Y, Z] as top-10,
    then recall@10 = 7/10 = 0.7 (missed H, I, J; returned wrong X, Y, Z)
    """
    approx_set = set(approx_results[:k])
    exact_set = set(exact_results[:k])
    return len(approx_set & exact_set) / k
```

Why recall and not precision? In a retrieval context, we care about **how many true nearest neighbors we found**, not about whether the approximately-nearest neighbors are "precise." If the exact top-10 includes document D at rank 8, and our ANN returns it at rank 12 (just outside top-10), that's a recall miss — but the document is still highly relevant. The user experience impact depends on how strict your ranking needs to be.

### Key Performance Metrics

| Metric | What It Measures | Typical Target | Why It Matters |
|--------|-----------------|----------------|----------------|
| Recall@10 | Accuracy of top-10 results | > 0.95 production | Determines result quality |
| QPS | Queries per second (throughput) | Application-dependent | Revenue capacity |
| P50 Latency | Median query time | < 10ms real-time | User experience |
| P99 Latency | Worst-case query time | < 50ms real-time | Tail latency matters for SLAs |
| Memory per vector | Index efficiency | Budget-dependent | Infrastructure cost |
| Build time | Time to construct index | < update frequency | Freshness constraints |

### Why P99 Matters More Than P50

In production, P50 latency tells you what "most" users experience, but **P99 tells you what your worst 1% of queries look like**. For an e-commerce site handling 1000 queries/second, P99 = 100ms means 10 queries per second are painfully slow. This is why search engineers obsess over tail latency, not just averages.

HNSW's latency distribution is typically well-behaved (tight P50-P99 spread), while IVF can have long tails when queries fall near cluster boundaries and require probing many cells.

## Algorithm Comparison at Scale

### Benchmark Data (1M vectors, 768-dim)

These benchmarks provide a realistic comparison of the major ANN algorithms on a standard dataset. The numbers come from real-world benchmarking (ann-benchmarks.com and FAISS benchmarks) and represent typical production-like conditions.

| Algorithm | Recall@10 | QPS (single thread) | Latency P50 | Memory | Build Time |
|-----------|----------|---------------------|-------------|--------|------------|
| Flat (exact) | 1.000 | ~50 | ~20ms | 3 GB | instant |
| IVF-Flat (nprobe=10) | 0.92 | ~2,000 | ~0.5ms | 3 GB | 2 min |
| IVF-Flat (nprobe=64) | 0.98 | ~400 | ~2.5ms | 3 GB | 2 min |
| IVF-PQ (nprobe=32) | 0.85 | ~5,000 | ~0.2ms | 200 MB | 5 min |
| IVF-PQ (nprobe=128) | 0.92 | ~1,200 | ~0.8ms | 200 MB | 5 min |
| HNSW (ef=32) | 0.92 | ~3,500 | ~0.3ms | 3.5 GB | 15 min |
| HNSW (ef=128) | 0.98 | ~1,200 | ~0.8ms | 3.5 GB | 15 min |
| HNSW (ef=256) | 0.995 | ~500 | ~2ms | 3.5 GB | 15 min |
| HNSW (ef=512) | 0.999 | ~200 | ~5ms | 3.5 GB | 15 min |

### Key Observations

**HNSW dominates the recall-speed frontier**: At every recall level, HNSW achieves the best QPS. At recall=0.98, HNSW delivers ~1200 QPS vs IVF-Flat's ~400 QPS — 3x faster for the same accuracy. This is why HNSW is the default choice for in-memory vector search.

**IVF-PQ wins decisively on memory**: At 200 MB vs 3.5 GB, IVF-PQ uses **17x less memory** than HNSW. For billion-scale deployments where memory costs dominate, this difference is the deciding factor. You can fit 17x more vectors in the same hardware with IVF-PQ.

**Flat index is unbeatable for small data**: Below 100K vectors, exact search completes in <10ms. Adding an approximate index introduces complexity without meaningful benefit. Don't over-engineer small datasets.

**There's no "best" algorithm**: The right choice depends entirely on your combination of recall requirements, latency budget, memory constraints, and update frequency.

## Tuning the ef_search Knob (HNSW)

The `ef_search` parameter is HNSW's primary accuracy/speed dial at query time. It controls how many candidates are explored during the graph traversal at layer 0. Higher ef_search means wider exploration, which finds more of the true nearest neighbors but takes more time.

```
ef_search = 16:  Recall@10 ≈ 0.85, Latency ≈ 0.15ms  (fast but misses a lot)
ef_search = 32:  Recall@10 ≈ 0.92, Latency ≈ 0.3ms   (acceptable for many apps)
ef_search = 64:  Recall@10 ≈ 0.96, Latency ≈ 0.5ms   (good for production)
ef_search = 128: Recall@10 ≈ 0.98, Latency ≈ 1ms     (high quality search)
ef_search = 256: Recall@10 ≈ 0.995, Latency ≈ 2ms    (near-perfect)
ef_search = 512: Recall@10 ≈ 0.999, Latency ≈ 5ms    (virtually exact)
```

**Rule of thumb**: Set ef_search ≥ K (where K is the number of results requested). For production, 4-10× K provides a good recall/speed balance. For top-10 search, ef_search=64-128 is a common production setting.

**Important nuance**: The recall/latency curve has **diminishing returns**. Going from ef=32 to ef=128 (4x increase) improves recall from 0.92 to 0.98 (a significant 6% gain). Going from ef=256 to ef=512 (2x increase) improves recall from 0.995 to 0.999 (a barely noticeable 0.4% gain). Most of the "easy" recall is captured with small ef values; the last few percentage points are increasingly expensive.

## Tuning the nprobe Knob (IVF)

For IVF indexes, the `nprobe` parameter controls how many Voronoi cells (clusters) are searched at query time. Searching more clusters finds more candidates but requires more distance computations.

```
nprobe = 1:   Recall@10 ≈ 0.45 (useless — misses most neighbors)
nprobe = 4:   Recall@10 ≈ 0.75 (still too low for production)
nprobe = 8:   Recall@10 ≈ 0.85 (acceptable for fast, best-effort search)
nprobe = 32:  Recall@10 ≈ 0.95 (good production setting)
nprobe = 64:  Recall@10 ≈ 0.97 (high quality)
nprobe = 128: Recall@10 ≈ 0.99 (near-perfect, slower)
nprobe = 512: Recall@10 ≈ 0.999 (effectively exact within quantization error)
```

**Why nprobe=1 is so bad**: With nprobe=1, you only search the single closest cluster. If the query vector is near a cluster boundary, its true nearest neighbors might be in adjacent clusters. The probability of this happening is surprisingly high in high-dimensional spaces — up to 50% of true nearest neighbors can lie in neighboring cells.

**The nprobe/nlist interaction**: The effectiveness of nprobe depends on nlist (total number of clusters). With nlist=1024, nprobe=32 means searching 3.1% of clusters. With nlist=65536, nprobe=32 means searching only 0.05% of clusters — which may be insufficient. As a heuristic, nprobe should search at least 1% of clusters for reasonable recall.

## Memory vs Accuracy Tradeoff

Quantization reduces memory but introduces approximation error. This is a separate knob from the search parameters (ef_search, nprobe) — quantization affects accuracy even with exhaustive search over all quantized vectors.

### Quantization Levels

| Method | Bytes/vector (768d) | Recall Loss vs float32 | Memory (1B vecs) | Speed Impact |
|--------|---------------------|----------------------|-------------------|-------------|
| float32 (baseline) | 3,072 | — | 3,000 GB | Baseline |
| float16 | 1,536 | < 0.5% | 1,500 GB | 1.5x faster |
| int8 (scalar quant) | 768 | 2-5% | 750 GB | 3x faster |
| PQ (M=96, 8-bit) | 96 | 5-15% | 96 GB | 8x faster |
| Binary quantization | 96 | 10-30% | 96 GB | 32x faster |

The recall loss from quantization is **additive** with the loss from approximate search. If your HNSW index achieves 98% recall with float32, the same index with int8 quantization might achieve 95% recall, and with PQ it might achieve 88%. Planning must account for both sources of approximation.

### The Two-Phase Strategy: Oversample and Rescore

Production systems commonly use a **two-phase** approach to get compressed-memory costs with near-full-precision accuracy:

```python
def two_phase_search(query, compressed_index, full_vectors, k=10, oversample=10):
    """
    Phase 1: Search with compressed vectors (PQ/int8) → get top k×oversample
             candidates. This is fast and memory-efficient.

    Phase 2: Load full-precision vectors ONLY for the candidates,
             re-score using exact distance, return top-k.
    """
    # Phase 1: Fast approximate search (uses compressed index in RAM)
    candidates = compressed_index.search(query, k=k * oversample)  # top 100
    
    # Phase 2: Exact re-scoring (loads 100 full vectors from SSD — cheap)
    rescored = []
    for doc_id in candidates:
        full_vec = full_vectors.load(doc_id)  # From SSD: ~0.1ms per vector
        exact_score = cosine_similarity(query, full_vec)
        rescored.append((exact_score, doc_id))
    
    return sorted(rescored, reverse=True)[:k]  # top 10
```

This approach is powerful because Phase 1 finds roughly the right documents (with PQ's 85-90% recall@100), and Phase 2 fixes the ranking using full-precision vectors. The result is typically 98%+ recall@10 with PQ-level memory usage — the best of both worlds.

The SSD cost of Phase 2 is negligible: loading 100 vectors × 3KB each = 300KB from SSD ≈ 0.5ms on NVMe.

## Decision Framework

### Choose Your Algorithm Based On Your Constraints

The most common mistake is over-engineering small datasets or under-engineering large ones. Here's a practical decision tree:

```
Q: How many vectors?
  
  < 100K vectors?
    → Use Flat index (exact brute-force search)
    → It's fast enough. ANN adds complexity without benefit.
    → Even at 100K × 768 dims, flat search takes ~8ms.

  100K - 10M vectors, sufficient RAM?
    → Use HNSW with M=16, ef_construction=200
    → Tune ef_search at query time: start at 64, increase for more recall
    → This handles 99% of production use cases in this range

  100K - 10M vectors, RAM is limited?
    → Use HNSW with scalar quantization (int8)
    → 4x memory reduction, ~2% recall loss
    → Or IVF-Flat if you need even less memory

  10M - 1B vectors, need <5ms latency?
    → HNSW with scalar quantization + two-phase re-scoring
    → Or IVF-PQ with high nprobe + re-scoring

  10M - 1B vectors, can tolerate 10-20ms?
    → DiskANN (Vamana on SSD)
    → 15-50x less RAM than HNSW
    → See Chapter 6.5 for DiskANN details

  > 1B vectors?
    → Distributed HNSW across shards (each shard < 100M)
    → Or IVF-PQ on disk with two-phase search
    → Plan for $25K+/month infrastructure costs
```

### Application-Specific Recommendations

| Application | Typical Scale | Recall Target | Latency Target | Recommended |
|-------------|-------------|---------------|----------------|-------------|
| Product search | 1M-50M | > 0.95 | < 20ms | HNSW (ef=128) |
| Semantic document search | 100K-10M | > 0.90 | < 50ms | HNSW (ef=64) |
| RAG / LLM retrieval | 10K-1M | > 0.98 | < 100ms | HNSW (ef=256) or flat |
| Image similarity | 10M-1B | > 0.90 | < 10ms | IVF-PQ + rescore |
| Recommendation system | 100M-10B | > 0.85 | < 20ms | IVF-PQ, sharded |
| Duplicate detection | 1M-100M | > 0.99 | < 1sec | Flat on batches |

## Benchmarking Your Own System

Never trust generic benchmarks blindly — ANN performance varies significantly with data distribution. Clustered data behaves differently from uniformly distributed data, and your production data is almost certainly not uniform.

```python
def benchmark_ann_index(index, query_vectors, ground_truth, 
                        ef_values=[32, 64, 128, 256, 512]):
    """
    Benchmark recall vs latency across ef_search values.
    ground_truth is from brute-force search on the same data.
    """
    for ef in ef_values:
        index.ef_search = ef
        
        recalls, latencies = [], []
        for query, true_neighbors in zip(query_vectors, ground_truth):
            start = time.time()
            results = index.search(query, k=10)
            latencies.append(time.time() - start)
            
            recall = len(set(results) & set(true_neighbors)) / 10
            recalls.append(recall)
        
        print(f"ef={ef:4d}: recall@10={mean(recalls):.4f}, "
              f"latency P50={percentile(latencies, 50)*1000:.1f}ms, "
              f"P99={percentile(latencies, 99)*1000:.1f}ms")
```

Run this benchmark with **your production query distribution**, not random vectors. Queries that are common in your application may behave differently from uniformly sampled test queries.

## Key Takeaways

1. **Recall vs latency** is the fundamental ANN tradeoff, controlled by ef_search (HNSW) or nprobe (IVF). Both show diminishing returns — the last 1% of recall costs disproportionately more compute.
2. **HNSW dominates** the recall/latency frontier for in-memory search, but costs ~17x more memory than IVF-PQ
3. **IVF-PQ** is the memory champion: 32x compression enables billion-scale search in manageable RAM, but recall@10 drops to 85-92% without re-scoring
4. **Two-phase search** (compressed retrieval + exact re-scoring of top candidates) is the production standard for balancing memory and accuracy
5. **Don't over-engineer small data**: Below 100K vectors, flat (exact) search completes in <10ms. Use approximate methods only when exact search is too slow.
6. **Benchmark on YOUR data**: Published benchmarks are directional guides, not guarantees. Data distribution, query patterns, and hardware all affect real-world performance.
7. **P99 matters more than P50**: For production SLAs, tail latency determines user experience for your worst 1% of queries
