# Chapter 5.6: Early Termination and WAND - Deep Research

## Overview

Early termination is a critical optimization for top-k retrieval in search engines. Instead of exhaustively scoring every document that matches a query, we can skip documents that provably cannot make it into the top-k results. The WAND (Weak AND) algorithm and its variants are the state-of-the-art techniques for this optimization.

---

## The Problem: Exhaustive Scoring is Expensive

For a query like `machine learning tutorial`:
- Boolean retrieval finds ALL documents containing any term (potentially millions)
- Each match needs to be scored (BM25 computation per document)
- Only top 10-20 results are shown to users

**The insight**: We only need the TOP-K documents, not ALL matches ranked.

---

## WAND Algorithm (Weak AND)

### Core Concept

WAND uses **upper-bound scores** to skip documents that cannot possibly rank in the top-k.

### Key Data Structures

1. **Posting Lists**: Sorted by docID, augmented with term frequencies
2. **Upper Bound Scores**: For each term t, precompute `UB(t)` = maximum score contribution t can have for ANY document
3. **Threshold θ**: The score of the current k-th best document found so far

### Upper Bound Computation

```python
def compute_upper_bound(term, index):
    """
    Upper bound = max TF score * IDF
    The max TF is found in the document with highest term frequency
    """
    idf = log(N / df(term))
    max_tf = max(tf(term, doc) for doc in posting_list(term))
    return bm25_tf_component(max_tf) * idf
```

### The Pivot Algorithm

```python
def wand_algorithm(query_terms, k):
    # Initialize iterators for each term's posting list
    iterators = [PostingIterator(term) for term in query_terms]
    
    # Priority queue for current docIDs (sorted by docID)
    # Each iterator tracks its current docID
    
    # Threshold = score of k-th best document found
    threshold = 0
    top_k_heap = MinHeap(capacity=k)
    
    while not all_exhausted(iterators):
        # Sort iterators by current docID
        iterators.sort(key=lambda it: it.current_doc_id())
        
        # Find PIVOT: smallest docID where cumulative UB >= threshold
        cumulative_ub = 0
        pivot_idx = -1
        
        for i, it in enumerate(iterators):
            cumulative_ub += upper_bounds[it.term]
            if cumulative_ub >= threshold:
                pivot_idx = i
                pivot_doc = it.current_doc_id()
                break
        
        if pivot_idx == -1:
            # No more documents can beat threshold
            break
        
        # Check if all iterators up to pivot point to same doc
        if all(iterators[j].current_doc_id() == pivot_doc for j in range(pivot_idx + 1)):
            # Full evaluation: compute exact score
            score = compute_full_score(pivot_doc, query_terms)
            if score > threshold:
                top_k_heap.push((score, pivot_doc))
                if len(top_k_heap) == k:
                    threshold = top_k_heap.peek()[0]  # Update threshold
            
            # Advance all iterators pointing at pivot_doc
            for j in range(pivot_idx + 1):
                iterators[j].next()
        else:
            # Not all aligned: advance lagging iterators to pivot_doc
            for j in range(pivot_idx):
                iterators[j].advance_to(pivot_doc)
    
    return top_k_heap.to_sorted_list()
```

### Visual Example

```
Query: "machine learning" (2 terms)
UB(machine) = 5.2, UB(learning) = 4.8
Current threshold θ = 7.5

Posting Lists (sorted by docID):
Machine:  [3, 7, 12, 25, 31, ...]
Learning: [5, 12, 18, 25, 40, ...]

Iteration 1:
  Machine points to:  doc 3
  Learning points to: doc 5
  
  Cumulative UB check for pivot:
    Just Machine: 5.2 < 7.5 (not enough)
    Machine + Learning: 5.2 + 4.8 = 10.0 >= 7.5 ✓
    
  Pivot = doc 5 (first doc where cumulative UB >= θ)
  
  But Machine (doc 3) != Learning (doc 5)
  → Advance Machine to doc 5 (but it goes to doc 7)
  
Iteration 2:
  Machine points to:  doc 7
  Learning points to: doc 5
  
  → Advance Learning to doc 7 (goes to doc 12)
  
... continues until convergence
```

### Why It Works

1. **Safe skipping**: If cumulative UB of terms < threshold, those documents CANNOT beat top-k
2. **Pivot selection**: The pivot is the earliest document that MIGHT qualify
3. **Efficient advancement**: Lagging iterators jump directly to pivot (using binary search in posting list)

## Block-Max WAND (BMW)

### Problem with Basic WAND

Upper bounds are computed **globally** (across entire posting list). This is often too loose:
- Max TF might be 1000 in one document
- Most documents have TF of 1-5
- Global UB is dominated by outliers

### Block-Max Enhancement

Divide posting lists into **blocks** (e.g., 128 documents each) and store **per-block maximum scores**.

```
Posting List for "machine":
Block 0 (docs 0-127):    maxScore = 3.2
Block 1 (docs 128-255):  maxScore = 2.1
Block 2 (docs 256-383):  maxScore = 5.2  ← global max
Block 3 (docs 384-511):  maxScore = 1.8
...
```

### Block-Max WAND Algorithm

```python
def block_max_wand(query_terms, k):
    iterators = [BlockMaxIterator(term) for term in query_terms]
    threshold = 0
    top_k = MinHeap(capacity=k)
    
    while not all_exhausted(iterators):
        iterators.sort(key=lambda it: it.current_doc_id())
        
        # Use BLOCK-LEVEL max scores for pivot computation
        cumulative_block_ub = 0
        pivot_idx = -1
        
        for i, it in enumerate(iterators):
            # Get max score for CURRENT BLOCK, not global
            cumulative_block_ub += it.current_block_max_score()
            if cumulative_block_ub >= threshold:
                pivot_idx = i
                break
        
        if pivot_idx == -1:
            # Can we skip to a more promising block?
            # Find next block that might qualify
            skip_to = find_next_promising_block(iterators, threshold)
            if skip_to is None:
                break
            advance_all_to(iterators, skip_to)
            continue
        
        # ... rest similar to WAND
```

### Performance Gains

Block-Max WAND provides **2-10x speedup** over basic WAND because:
- Tighter upper bounds → more aggressive skipping
- Can skip ENTIRE BLOCKS when their max scores are too low
- Block size is tuned for cache efficiency (often 128 docs)

## MaxScore Algorithm

An alternative to WAND, often preferred for OR queries.

### Core Idea

Partition query terms into:
1. **Essential terms**: High UB, must be checked for every candidate
2. **Non-essential terms**: Low UB, only checked when a candidate is found via essential terms

### Algorithm

```python
def max_score(query_terms, k):
    # Sort terms by ascending upper bound
    terms = sorted(query_terms, key=lambda t: upper_bound(t))
    
    threshold = 0
    top_k = MinHeap(capacity=k)
    
    # Partition: non-essential terms have cumulative UB < threshold
    # Essential terms are the rest
    
    while True:
        # Determine partition point
        cumulative = 0
        essential_start = 0
        for i, term in enumerate(terms):
            if cumulative >= threshold:
                essential_start = i
                break
            cumulative += upper_bound(term)
        
        # Iterate through essential terms only
        # Non-essential terms (0 to essential_start-1) are "cheap"
        
        candidate = find_next_candidate_via_essential_terms(...)
        
        if candidate is None:
            break
        
        # Score candidate using ALL terms
        score = compute_full_score(candidate, query_terms)
        
        if score > threshold:
            top_k.push((score, candidate))
            if len(top_k) == k:
                threshold = top_k.peek()[0]
                # Re-partition after threshold update
```

### When to Use MaxScore vs WAND

| Scenario | Preferred Algorithm |
|----------|---------------------|
| Few query terms (2-5) | WAND |
| Many query terms (5+) | MaxScore |
| High k (100+) | MaxScore |
| Low k (10-20) | WAND |
| Conjunctive (AND) queries | WAND |
| Disjunctive (OR) queries | MaxScore |

## Early Termination Strategies

### 1. terminate_after (Elasticsearch)

```json
{
  "query": { "match": { "content": "search engine" } },
  "terminate_after": 1000
}
```

**Per-shard termination**: Stop after collecting N documents per shard.

**Caveats**:
- Results may be incomplete
- Inconsistent with sorting (top docs might be missed)
- Response includes `terminated_early: true` flag

### 2. Timeout-Based Termination

```json
{
  "query": { "match": { "content": "search engine" } },
  "timeout": "100ms"
}
```

Return best results found within time limit.

### 3. Document-at-a-Time (DAAT) vs Score-at-a-Time (SAAT)

- **DAAT**: Score documents one by one (WAND uses this)
- **SAAT**: Score all documents for one term, then next term (less common for early termination)

## Skip Pointers in Posting Lists

### Purpose

Accelerate `advance(targetDoc)` operations in posting list traversal.

### Structure

```
Posting List: [3, 7, 12, 25, 31, 48, 55, 67, 82, 91, ...]
                ↓        ↓              ↓
Skip Pointers: [3→25]   [25→55]        [55→91]
```

### Optimal Skip Interval

**Heuristic**: For posting list of length P, use √P evenly-spaced skip pointers.

```python
skip_interval = int(sqrt(len(posting_list)))
skip_pointers = posting_list[::skip_interval]
```

### Trade-off

- More skip pointers → faster skipping
- More skip pointers → more storage overhead
- √P balances search time and space

## Implementation in Lucene

### Block-Max Index Structure

Lucene stores posting lists in blocks of 128 documents:
- Each block stores delta-encoded docIDs
- Each block stores max impact score for that block
- Enables WAND/MaxScore algorithms

### Lucene 8+ Evolution

| Version | Default Algorithm |
|---------|-------------------|
| Lucene 7 | Exhaustive scoring |
| Lucene 8 | Block-Max WAND |
| Lucene 9+ | Block-Max MaxScore (for disjunctions) |

### Key Classes

- `Weight.scorer()` → Returns `Scorer` with WAND/MaxScore support
- `BulkScorer` → Block-level scoring for efficiency
- `ImpactsDISI` → Impacts-aware DocIdSetIterator

## Performance Considerations

### When Early Termination Works Best

1. **High selectivity queries**: Many terms, but few top results needed
2. **Skewed score distributions**: Few documents have high scores
3. **Large posting lists**: More opportunities to skip

### When It Underperforms

1. **Low k values with uniform scores**: Hard to establish useful threshold
2. **Single-term queries**: No intersection opportunities
3. **Mandatory filters**: Reduces posting list sizes anyway

### Benchmarks (typical)

| Query Type | Speedup vs Exhaustive |
|------------|----------------------|
| 2-term OR | 2-3x |
| 3-term OR | 3-5x |
| 5-term OR | 5-10x |
| Phrase query | 1-2x |

---

## Key Takeaways

1. **WAND** uses upper-bound scores and pivot selection to safely skip non-competitive documents
2. **Block-Max WAND** provides tighter bounds by storing per-block max scores
3. **MaxScore** is preferred for high-k or many-term queries
4. **Skip pointers** accelerate posting list traversal within these algorithms
5. **terminate_after** is a simpler but less precise early termination
6. Modern Lucene (8+) uses Block-Max algorithms by default
