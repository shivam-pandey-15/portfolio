# Chapter 7.5: Evaluation Metrics for Embedding Training

## Executive Summary

Training a custom embedding model is only useful if you can prove it improves retrieval quality in a way that matters. This sounds obvious, but many teams still rely on training loss, anecdotal examples, or a handful of demo queries. That is not evaluation. In search, the model must be measured on ranked retrieval behavior using a held-out set of realistic queries and trustworthy relevance judgments.

The central challenge is that embedding evaluation sits between machine learning and search engineering. It is not enough to ask, "Does the positive pair score high?" You must ask whether the system retrieves the right candidates at the right depth, with the right ordering, under the same constraints that production imposes.

This chapter explains the core retrieval metrics, how to build a reliable benchmark, why offline metrics can still mislead you, and how to connect model evaluation to business outcomes.

---

## What We Are Actually Evaluating

### Retrieval, Not Classification

An embedding model does not produce a binary yes or no answer. It produces a ranked neighborhood. Evaluation therefore asks questions like:

- Did the right document appear in the top 10?
- How early did the first relevant result appear?
- Were multiple relevant results surfaced?
- Did the model preserve recall for downstream reranking?

This is why standard classification metrics such as accuracy are usually unhelpful here. Search quality depends on rank order and candidate coverage, not only on whether some thresholded decision was correct.

### Candidate Generation vs Final Ranking

You must also be clear about what layer you are evaluating. For dense retrieval, there are typically two different goals:

1. First-stage retrieval quality.
2. End-to-end ranking quality after reranking.

If the dense model is a candidate generator, Recall@K often matters more than fine-grained ordering inside the top few results. If the dense model is the final ranker, metrics like nDCG@10 and MRR@10 matter more directly.

## Core Retrieval Metrics

## nDCG@K

### What It Measures

Normalized Discounted Cumulative Gain evaluates how much relevance is packed into the top of the ranking, while discounting lower positions logarithmically. It is one of the most useful metrics because it handles:

- multiple levels of relevance
- sensitivity to rank position
- comparison across queries with different numbers of relevant documents

The model gets more credit for placing a highly relevant document at rank 1 than rank 8.

### Why Search Teams Like It

nDCG is often the best single summary metric for production ranking work because it reflects both ordering and graded usefulness.

```python
import math

def dcg(relevances):
    total = 0.0
    for rank, rel in enumerate(relevances, start=1):
        total += rel / math.log2(rank + 1)
    return total

def ndcg(relevances, ideal_relevances):
    return dcg(relevances) / dcg(ideal_relevances)
```

### When It Is Especially Valuable

nDCG is strong when:

- queries may have several useful results
- you have graded labels such as 0, 1, 2, 3
- you care about top-of-page quality, not just recall

## MRR@K

### What It Measures

Mean Reciprocal Rank looks only at the first relevant result. If the first relevant document appears at rank 1, the score is `1.0`. If it appears at rank 5, the score is `0.2`.

```python
def reciprocal_rank(results, relevant_set):
    for rank, doc_id in enumerate(results, start=1):
        if doc_id in relevant_set:
            return 1.0 / rank
    return 0.0
```

### Why It Is Useful

MRR is excellent for known-item retrieval and question-answering style search where users usually need one correct answer fast.

Examples:

- finding a specific support article
- retrieving a precise configuration page
- finding the right legal clause
- matching an exact troubleshooting runbook

### Its Main Limitation

MRR ignores all relevant results after the first one. A ranking with one good result followed by nine terrible ones can still look strong under MRR.

---

## Recall@K

### What It Measures

Recall@K asks whether relevant documents appeared anywhere in the top K results. It does not care much about exact ordering inside that window.

This is crucial when dense retrieval is just the first phase of a multi-stage search system. If the downstream reranker only sees the top 100 candidates, then missing the right document entirely is catastrophic.

### Why Retrieval Engineers Care About It

A retriever can have mediocre nDCG@10 but still be very useful if Recall@100 is high enough for a stronger reranker to fix the final ordering. That is why candidate-generation systems often optimize for recall-heavy metrics first.

### Typical Uses

- dense first-stage retrieval
- ANN hyperparameter tuning
- chunking comparisons
- evaluation of hybrid candidate unions

---

## Precision@K and MAP

### Precision@K

Precision@K measures how many of the top K results are relevant. It is simple and intuitive, but it does not distinguish whether the relevant items were at positions 1, 2, or 10.

### MAP

Mean Average Precision rewards systems that rank relevant documents early across the whole ranking. It is still used in IR research, though many production teams lean more heavily on nDCG and recall-based metrics because they map more directly to their retrieval pipeline choices.

## Building a Reliable Evaluation Set

### The Gold Standard: Held-Out Query Judgments

A benchmark is only as good as its labels. For embedding evaluation, you need a set of real or realistic queries and judged relevant documents that the model did not see during training.

Good evaluation data should be:

- representative of production query mix
- separated from training by time or intent family
- balanced across head and tail queries
- large enough to stabilize comparisons

### How Big Should It Be?

There is no universal magic number, but useful ranges often look like this:

| Use Case | Rough Size |
|----------|------------|
| Early prototype | 200 to 500 judged queries |
| Serious model comparison | 1,000 to 5,000 judged queries |
| High-stakes production system | 5,000+ with slice coverage |

The important property is not just volume. It is diversity and trustworthiness.

### Label Sources

Good labels can come from:

- expert annotators
- vetted click-derived judgments
- curated FAQ mappings
- support workflow resolution data
- human review of model outputs

Weak labels are sometimes unavoidable, but you should know how weak they are.

## Evaluation Slices Matter More Than a Single Number

### Why Aggregate Metrics Hide Failures

A model can improve overall nDCG while getting worse on the query types your product cares most about. That is why mature teams evaluate slices, not just averages.

Useful slices include:

- head vs tail queries
- exact entity queries
- semantic paraphrase queries
- multilingual queries
- short queries vs long natural-language questions
- newly added documents or categories

### Example

Suppose a model improves overall nDCG@10 by 3 percent, but exact product-code retrieval drops by 10 percent. For a B2B catalog, that may be unacceptable even though the global metric looks better.

Search systems are shaped by these local trade-offs.

## Offline Evaluation Pitfalls

### False Negatives in Judgment Sets

Your benchmark may treat an unjudged document as irrelevant even when it is actually a good answer. This happens constantly in search because human labeling coverage is limited.

Result:

- the model retrieves a useful unseen document
- the metric penalizes it as wrong

This is one reason offline metrics can understate real quality gains.

### Benchmark Leakage

If training pairs and evaluation queries overlap too closely, quality numbers inflate. Leakage can happen through:

- exact query duplication
- paraphrase variants
- same documents appearing with near-identical query families
- time-based leakage from future clicks

### Evaluation on the Wrong Unit

Sometimes teams evaluate document-level relevance while the product actually operates on chunks, products, tickets, or entities. If the benchmark unit does not match the production retrieval unit, metric interpretation becomes fuzzy.

### Ignoring ANN Effects

A model may look excellent when scored against brute-force similarity but degrade once deployed in an approximate nearest neighbor index with aggressive recall-latency settings. Realistic evaluation should include the retrieval stack you actually serve.

## Connecting Offline Metrics to Online Outcomes

### Why Correlation Is Imperfect

Offline metrics are necessary, but they are not the same as user value. A model can improve nDCG while barely moving business metrics if:

- the bad results were on low-traffic queries
- the ranking change was too small to affect behavior
- the UI hides the improvement
- the reranker or filter layer cancels the gain

### Still, Offline Metrics Matter

They are essential because they let you iterate cheaply and safely before running live experiments. Without them, every model decision becomes guesswork.

A good mental model is:

- offline metrics decide what deserves an A/B test
- online metrics decide what deserves deployment

### Business Metrics to Pair With Retrieval Metrics

Depending on the product, track:

- click-through rate
- reformulation rate
- conversion after search
- time to resolution
- zero-result escape rate
- ticket deflection
- revenue per search

The point is not to replace IR metrics with business metrics. It is to connect them.

## Practical Evaluation Workflow

### A Strong Default Loop

1. Evaluate the baseline model on held-out queries.
2. Compare candidate retrieval metrics such as Recall@50 and Recall@100.
3. Compare ranking-sensitive metrics such as nDCG@10 and MRR@10.
4. Inspect slice-level changes.
5. Run brute-force and ANN-backed comparisons.
6. Only then promote a model to online testing.

### Example Evaluation Skeleton

```python
def evaluate_model(model, queries, corpus, qrels, k=10):
    metrics = {
        "ndcg": 0.0,
        "mrr": 0.0,
        "recall": 0.0,
    }

    for q in queries:
        ranked = retrieve(model, q, corpus, top_k=100)
        metrics["ndcg"] += ndcg_for_query(ranked[:k], qrels[q.id])
        metrics["mrr"] += mrr_for_query(ranked[:k], qrels[q.id])
        metrics["recall"] += recall_for_query(ranked[:100], qrels[q.id])

    for key in metrics:
        metrics[key] /= len(queries)

    return metrics
```

This is simple, but it captures the key principle: evaluation should mirror the task structure you care about.

## Reading Metric Changes Correctly

### Not Every Delta Is Meaningful

Small metric changes can be noise, especially on tiny evaluation sets. That is why teams often use:

- bootstrap confidence intervals
- paired significance testing
- repeated benchmark runs
- fixed evaluation slices over time

### What a Good Improvement Looks Like

A convincing model improvement usually has several properties:

- metric lift is stable across reruns
- head and tail slices do not regress badly
- exact-match or entity-sensitive queries are protected
- ANN deployment reproduces the offline gain
- qualitative review agrees with the numbers

The best evaluation combines quantitative rigor with qualitative failure analysis.

---

## Key Takeaways

1. Embedding evaluation is about ranked retrieval behavior, not classification accuracy or training loss.
2. nDCG, MRR, and Recall each answer different questions; good teams use all three rather than relying on a single score.
3. A trustworthy held-out benchmark with realistic queries and reliable labels is more important than any specific metric formula.
4. Aggregate metrics can hide serious regressions, so slice-based evaluation is essential.
5. Offline metrics should gate experimentation, but online metrics must decide final deployment value.
