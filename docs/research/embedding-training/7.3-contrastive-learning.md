# Chapter 7.3: Contrastive Learning for Embeddings

## Executive Summary

Contrastive learning is the engine behind most modern embedding models used in search. Instead of asking the model to predict the next token, we ask it to learn geometry: bring related texts together and push unrelated texts apart. That sounds simple, but the details matter enormously. Batch construction, temperature, negative quality, pooling, normalization, and loss selection all shape the embedding space that your retrieval system will live inside.

For search engineers, the key conceptual shift is this: retrieval training is not primarily about classification accuracy. It is about neighborhood structure. A good embedding model does not merely label a pair as similar. It organizes the space so that approximate nearest neighbor search can recover the right candidates quickly and consistently.

This chapter explains how contrastive learning works, why bi-encoders dominate first-stage retrieval, and where training runs typically succeed or fail in practice.

---

## The Geometric Objective

### What the Model Is Trying to Learn

Suppose we embed a query and a document into vectors of equal dimension. Contrastive learning defines a training signal over those vectors:

- positive pairs should have high similarity
- negative pairs should have low similarity

The model is rewarded when the true document is closer to the query than competing documents. In search terms, it is learning a retrieval-friendly space where nearest neighbors approximate relevance.

### Why This Is Better Than Generic Similarity Training

A lot of early semantic methods optimized generic sentence similarity. Search is stricter. The model must learn retrieval order, not just broad semantic resemblance. A query like "best budget monitor for coding" should not merely land near "computer displays." It should land nearer to documents that satisfy the purchase intent than to docs about monitor calibration, gaming displays, or laptop screen repair.

That is why dense retrieval training usually uses search-style positives and negatives rather than only paraphrase datasets.

## Why Bi-Encoders Are the Standard for Retrieval

### Bi-Encoder Structure

In a bi-encoder, the query and the document are encoded separately:

```python
q_vec = query_encoder(query_text)
d_vec = doc_encoder(document_text)
score = cosine_similarity(q_vec, d_vec)
```

Often the query and document encoder share weights, which is the classic Siamese setup. Sometimes they are asymmetrically tuned, especially when queries are extremely short and documents are long or stylistically different.

### Why Bi-Encoders Win Operationally

Bi-encoders are popular because document embeddings can be precomputed once and stored in a vector index. At query time, you only encode the query and retrieve nearest neighbors.

That gives them the right production shape for:

- large corpora
- low-latency serving
- ANN retrieval
- batching document indexing offline

Cross-encoders are more accurate for reranking, but they require joint query-document inference at runtime and are too expensive for candidate generation at scale.

### Symmetric vs Asymmetric Encoders

Many open-source retrieval models use a shared encoder because it is simpler and usually good enough. But some systems intentionally treat query and document text differently:

- Queries are short and noisy.
- Documents are long and structured.
- Query intent is often underspecified.
- Documents may require stronger compression.

This leads to asymmetric designs where prompts, pooling, or even learned towers differ between query and document inputs. The more your production inputs differ in form, the more attractive asymmetric training becomes.

## InfoNCE and Multiple Negatives Ranking Loss

### The Core Loss

The most common objective in modern embedding training is a form of InfoNCE or Multiple Negatives Ranking Loss. In a batch of aligned query-positive pairs:

- query `i` should match document `i`
- documents from all other pairs act as negatives

```python
import torch
import torch.nn.functional as F

def mnr_loss(query_embs, doc_embs, temperature=0.05):
    query_embs = F.normalize(query_embs, dim=1)
    doc_embs = F.normalize(doc_embs, dim=1)

    scores = torch.matmul(query_embs, doc_embs.T) / temperature
    labels = torch.arange(scores.size(0), device=scores.device)
    return F.cross_entropy(scores, labels)
```

This loss is effective because every batch creates many implicit negatives without storing them explicitly on disk.

### Why Temperature Matters

Temperature controls how sharply the model focuses on the hardest competitors.

- Higher temperature: smoother distribution, softer penalties
- Lower temperature: sharper distribution, harder penalties for close negatives

If temperature is too high, the model may not separate close confusions strongly enough. If it is too low, training can become unstable or over-sensitive to noise and false negatives.

### Softmax View of the Problem

A useful mental model is that each row of the similarity matrix is a multiple-choice question. For query `q_i`, the model is asked: which document in this batch is the correct match? The correct answer is the positive on the diagonal, and every off-diagonal item is a distractor.

That framing matters because retrieval is inherently comparative. The model is not just learning that a query and document are related in isolation. It is learning that the positive should beat every nearby alternative.

## The Importance of Batch Size

### Bigger Batches Mean More Negatives

With in-batch negatives, a batch of 32 gives each query 31 negatives. A batch of 1024 gives each query 1023 negatives. That usually improves learning because the model sees a richer competitive set.

This is why practitioners say batch size is one of the strongest levers in contrastive learning. It changes how hard the task is.

### But Bigger Is Not Free

Large batches introduce costs:

- more VRAM
- slower communication across devices
- more false negatives in narrow domains
- more engineering complexity

Teams often use gradient accumulation, distributed training, or GradCache-style techniques to simulate large effective batches without holding the entire autograd graph in memory.

### Why Very Large Batches Can Backfire

In broad web corpora, more negatives are often helpful. In narrow domains, giant batches can quietly hurt because many items are semantically adjacent. If you train on an enterprise corpus where every document is about payroll, HR policy, or identity management, then several random in-batch documents may actually be acceptable results.

That creates accidental false negatives. So "bigger batch = better" is only true when the negative pool is trustworthy enough.

## Pooling, Normalization, and Similarity

### Pooling Creates the Final Sentence Vector

Transformer encoders output token-level vectors. Retrieval systems need one vector per query or document, so we pool.

Common pooling choices:

| Pooling | What It Does | Typical Behavior |
|---------|---------------|------------------|
| CLS | Use special first token | Fast, sometimes weaker |
| Mean pooling | Average token vectors | Strong default for retrieval |
| Max pooling | Take element-wise max | Less common, can be noisy |

Mean pooling is widely used because it tends to produce stable retrieval embeddings across domains.

### Normalization Matters

Most dense retrieval systems L2-normalize embeddings before similarity comparison. This turns dot product into cosine similarity and stabilizes score scales across examples.

Without normalization, vector magnitude can dominate ranking in undesirable ways. With normalization, direction becomes the main carrier of semantic information.

### Dot Product vs Cosine Similarity

If all vectors are normalized, dot product and cosine similarity are equivalent. That is why many production systems can use inner-product ANN indexes while still reasoning conceptually in cosine space. The important point is consistency: train, index, and evaluate with the same similarity assumptions.

## Beyond Vanilla InfoNCE

### Triplet Loss

Triplet loss uses an explicit anchor, positive, and negative:

```python
loss = max(0, sim(q, neg) - sim(q, pos) + margin)
```

This is intuitive but usually less efficient than in-batch contrastive losses because it depends heavily on the quality of the chosen negative and does not exploit the whole batch as naturally.

### MarginMSE and Distillation-Style Losses

When you have a teacher model, reranker, or human score, you can train the embedding model to preserve score gaps:

- Query-document A should score 0.9
- Query-document B should score 0.4
- The embedding model learns the relative margin

This can work well for distilling a stronger but slower reranker into a fast dense retriever.

### Supervised Contrastive Variants

Some setups allow multiple positives per query or use class-aware groupings. These objectives can better reflect domains where several documents are valid for a single information need. For knowledge bases, support articles, and policy search, that is often closer to reality than the "one query, one document" assumption.

### Multi-Stage Training

A common recipe is:

1. Start with easy positives and in-batch negatives.
2. Add mined hard negatives.
3. Distill from a stronger reranker or teacher.
4. Fine-tune on the most important domain slices.

This curriculum often produces a more stable embedding space than jumping straight into the hardest possible training setup.

## Hard Negatives in the Loss

### Why They Improve Quality

Hard negatives make the denominator of the contrastive objective meaningful. If all negatives are obviously unrelated, the model can win without learning fine distinctions.

Important hard negative types:

- lexical lookalikes
- same-category but wrong-entity docs
- top ANN retrieval errors from current model
- top BM25 results skipped by users

These negatives teach the model the difference between "close but wrong" and "actually right."

### Why They Can Also Break Training

If your hard negatives contain many false negatives, the model gets contradictory signals. It is told to push away documents that users would actually consider relevant. That can flatten or distort local neighborhoods.

This is one reason training runs sometimes show decreasing loss but worse retrieval quality. The model is optimizing the labels you provided, not the reality you wanted.

### Online Mining vs Offline Mining

Offline mining is simple: precompute hard negatives once and train on them. Online mining is more adaptive: periodically refresh hard negatives using the latest model checkpoint. Online mining often yields stronger negatives later in training, but it adds infrastructure complexity and can create instability if refreshed too aggressively.

## Common Failure Modes

### 1. Representation Collapse

In severe cases, embeddings drift toward a narrow region of space and lose useful discrimination. Symptoms include:

- similarity scores bunch together
- ANN recall worsens
- nearest neighbors become generic

Causes include poor negatives, extreme learning rates, and unstable optimization.

### 2. False Negative Saturation

In narrow domains, large in-batch negatives can create many accidental collisions. If every document in the batch is about a similar product line or technical topic, several "negatives" may actually be relevant alternatives.

### 3. Shortcut Learning

The model learns superficial cues:

- brand names
- document length
- formatting markers
- category headers

It appears to improve offline but fails to generalize.

### 4. Overfitting to Training Query Style

If your training data contains only polished expert queries, the model may fail on messy real-user fragments. Query diversity matters as much as document diversity.

### 5. Embedding Space Drift Across Versions

A model can improve according to offline metrics yet produce a vector space that is poorly calibrated with your existing index, ANN hyperparameters, or reranking assumptions. That is not a theoretical issue. It is a common production issue when teams swap checkpoints without revalidating the rest of the stack.

## A Practical Training Recipe

### A Strong Default Stack

For many teams, a robust starting point looks like this:

1. Start from a proven base embedding model.
2. Use mean pooling and L2 normalization.
3. Train with Multiple Negatives Ranking Loss.
4. Begin with in-batch negatives.
5. Add BM25 or model-mined hard negatives later.
6. Evaluate frequently on a held-out search benchmark.

### Example Skeleton

```python
for batch in train_loader:
    q = encoder(batch["query_tokens"])
    d = encoder(batch["doc_tokens"])

    q = mean_pool(q, batch["query_mask"])
    d = mean_pool(d, batch["doc_mask"])

    loss = mnr_loss(q, d, temperature=0.05)

    loss.backward()
    optimizer.step()
    optimizer.zero_grad()
```

This is not the whole system, but it captures the core. Most practical complexity comes from data preparation, batching, negatives, and evaluation rather than from the three lines of math.

### Practical Hyperparameters to Watch

| Lever | Typical Range | Why It Matters |
|------|---------------|----------------|
| Learning rate | `1e-5` to `2e-4` | Too high destabilizes geometry |
| Temperature | `0.02` to `0.1` | Controls how sharply negatives are penalized |
| Batch size | `64` to `8192+` effective | Determines negative volume |
| Max sequence length | `128` to `512` | Impacts cost and document coverage |
| Hard negative count | `1` to `20` per query | Improves fine-grained separation |

These values are not universal defaults, but they are useful search ranges.

## How to Think About Quality During Training

### Loss Curves Are Not Enough

A falling contrastive loss does not guarantee better retrieval. You must evaluate retrieval metrics on held-out queries throughout training.

Good practice:

- track nDCG@10
- track Recall@50 or Recall@100
- inspect fixed failure slices
- compare head vs tail queries
- compare exact-entity queries separately from semantic queries

The right checkpoint is not always the one with the lowest training loss.

### Retrieval Is an Ecosystem Metric

Sometimes the embedding model improves in isolation but production quality does not. Reasons include:

- ANN index parameters are too aggressive
- chunking strategy is poor
- metadata filters remove the best candidates
- reranker calibration is off

Always judge the model both alone and inside the real retrieval stack.

---

## Key Takeaways

1. Contrastive learning trains embedding geometry by pulling relevant texts together and pushing irrelevant texts apart.
2. Bi-encoders dominate retrieval because they let you precompute document vectors and perform ANN search efficiently at runtime.
3. InfoNCE or Multiple Negatives Ranking Loss is the standard dense retrieval objective because it turns each batch into a competitive ranking problem.
4. Batch size, temperature, pooling, normalization, and negative quality strongly influence the learned space and often matter more than architectural novelty.
5. A successful training run is measured by held-out retrieval quality, not just lower loss.
